# TBX Compiler Design Principles

> このファイルは TBX 設計書の一部です。`blueprint.md`（VM・辞書のアーキテクチャ）と `blueprint-language.md`（コア言語仕様）と合わせて参照してください。

## この文書の目的

この文書は、TBX のコンパイル機構について、ソースコードだけでは読み取りにくい**設計原則・判断基準・避けるべき方向**を残すためのものとする。

- 現在の関数名、フィールド名、命令列、個別 primitive の一覧のような**実装事実**はソースコードを正とする
- この文書では、今後コンパイル機能を追加・変更するときに使う**判断基準**を優先して記述する
- 過去の実装履歴や issue ごとの経緯は必要最小限にとどめ、将来の設計判断に直接効く内容を残す

## コンパイラの役割

TBX のコンパイラは、BASIC 風の表面構文を、その都度専用実装へ落とし込む層ではない。小さな VM コアと辞書モデルの上で、**構文を拡張可能な形で lower する層**として位置付ける。

ここで重視するのは次の点である。

- 表面構文は BASIC らしく保つ
- 内部では小さく直交的な compile-time 操作へ分解する
- 特定構文のためだけの特別扱いを増やさず、複数機能から再利用できる語彙へ寄せる

TBX におけるコンパイル機構は、単なるソース変換器ではなく、自己拡張可能な言語処理系の一部である。

## 小さなコアと標準ライブラリの境界

TBX では、構文機能を追加したいときに、最初から Rust 側へ専用実装を入れることを避ける。

- Rust 側には、複数の構文やライブラリ機能から再利用できる VM / compile-time VM 操作を置く
- 構文として便利だが高水準なものは、可能な限り TBX 側の IMMEDIATE ワードまたは標準ライブラリで組み立てる
- ある構文を 1 つ実現するためだけの専用 primitive は、最後の手段として扱う

この境界を保つ理由は明確である。

- コアを小さく保つと、仕様変更の影響範囲を局所化しやすい
- ライブラリ側で構文を試せるため、設計の試行錯誤を Rust 実装へ固定化しにくい
- 構文を支える低レベル語彙が明確になり、別の構文でも再利用しやすい

## IMMEDIATE ワードの設計方針

TBX では、構文拡張の基本単位を IMMEDIATE ワードとする。これは Forth 的な自己拡張性を保ちつつ、BASIC 風の表面構文を段階的に育てるためである。

IMMEDIATE ワードを中心に据える理由は次の通り。

- 新しい構文を TBX 自身で定義しやすい
- コンパイル時の操作を辞書上の語彙として表現できる
- Rust 側へ構文専用ロジックを埋め込まずに済む
- 「どの構文糖衣が、どの低レベル操作に支えられているか」を見通しやすい

そのため、構文追加時の第一選択肢は常に「IMMEDIATE ワードと既存 compile-time 語彙の組み合わせで表せないか」である。Rust 側の変更は、その表現が不自然または過剰に複雑になる場合に限って検討する。

## compile stack の設計方針

compile stack は、実行時の data stack とは別の責務を持つ。両者を混ぜないのは、単なる実装都合ではなく、意味論の境界を保つためである。

- data stack は実行時値の受け渡しと計算に使う
- compile stack はコンパイル時の一時状態、未解決地点、構文スコープ管理に使う

この分離が必要な理由は次の通り。

- 実行時値とコンパイル時メタ情報では寿命も検証方法も異なる
- 制御構造の対応関係や未解決参照を、実行時スタック規約に依存せず扱える
- コンパイル時エラーを、その場で局所的に報告しやすい

compile stack は「構文ごとの特例を積む場所」ではなく、複数の IMMEDIATE ワードが共有できる**汎用の compile-time 作業領域**として保つべきである。

## token stream の設計方針

字句トークン列は、実行時値とも compile stack の値とも別物として扱う。

- `Token` / `SpannedToken` は入力ソースの字句情報を表す
- `CompileEntry` はコンパイル中の内部状態を表す
- `Cell` は実行時の値空間を表す

この 3 つを混ぜない理由は明確である。

- 字句情報には位置や表記の責務がある
- コンパイル中の内部状態には未確定な構文情報や補助メタデータがある
- 実行時値は VM が評価・保存・受け渡しする対象である

便利だからといって lexer token をそのまま data stack へ流したり、compile stack の都合で `Cell` に本来の責務外の意味を背負わせたりすると、層の境界が崩れ、仕様の説明も実装の保守も難しくなる。

## backpatching の設計方針

前方参照の解決やジャンプ先の後埋めは、特定構文専用の裏技としてではなく、**汎用の compile-time 操作**として扱う。

重要なのは、IF 専用、WHILE 専用、FOR 専用といった個別ロジックを Rust 側へ増やすことではない。必要なのは、未解決位置を記録し、後で埋め戻すという一般形を安定して扱えることである。

この方針により、制御構造の追加時には次の順で考える。

1. 既存の compile stack 操作と backpatching 操作の組み合わせで表せるか
2. 表せない場合、その不足は特定構文の都合か、それとも他の構文でも再利用できる抽象の不足か
3. 追加するなら、個別構文の名前を知らない低レベル語彙として定義できるか

## Lexer / StatementReader / Compiler の責務分離

TBX では、入力処理の層を分けて考える。

- Lexer は文脈フリーな字句分解を担当する
- StatementReader は物理行から論理ステートメントを組み立てる
- Compiler は構文規則と compile-time 操作へ落とし込む

この分離を崩さない理由は次の通り。

- Lexer が構文文脈まで知り始めると、字句規則が局所的な都合で歪む
- StatementReader に論理ステートメント境界を集約すると、行継続やラベル判定の責務を一箇所に閉じ込められる
- Compiler は「何をどう lower するか」に集中できる

たとえば、行番号ラベル判定のような処理は、字句器の責務ではなく、論理ステートメント境界を知る層に置くべきである。これは文脈依存判定を必要最小限の層へ隔離するためである。

## USE / source include の設計方針

`USE` は import ではなく、include 型の compile-time directive として扱う。指定された TBX ソースファイルは、その `USE` が現れた時点で読み込まれ、外側インタプリタによって処理される。

この設計では、`USE` は通常の実行時 file I/O ではない。構文としてはユーザーに見えるコア言語層に属するが、実際のパス解決・ファイル読み込み・ソース実行はコンパイル・外側インタプリタ層の責務である。VM 実行コアは `USE` を知らない。

`USE` の基本方針は次の通りである。

- `USE "path"` は、指定ファイルの内容をその場で処理する include である
- 同じファイルを複数回 `USE` した場合、原則としてそのたびに読み込まれる
- 読み込み済みファイルのグローバルキャッシュは持たない
- 循環検出は「二重読み込み禁止」ではなく、現在の `USE` チェーン内での無限再帰防止である
- 現在ロード中の canonical path が再登場した場合は循環参照としてエラーにする
- 非循環でも過度に深い `USE` チェーンはネスト深度制限で止める

この仕様により、`USE` は C の `#include` に近い素朴な合成手段として振る舞う。ライブラリや利用者は、同じファイルを複数回 include した場合の定義・副作用を意識する必要がある。一方で、import 的な「一度だけ読み込む」「名前空間を作る」「公開/非公開を管理する」といった責務は、現在の `USE` には持たせない。

相対パス解決は、ソース読み込みの文脈に依存する。ファイル内のネストした `USE` は、そのファイル自身のディレクトリを基準に解決される。トップレベルの `USE` は、外側インタプリタが持つ基準ディレクトリがあればそれを使い、なければ実行環境の既定に従う。この挙動により、ライブラリファイルは自身の隣接ファイルを相対パスで参照できる。

今後 import 的な機能を導入する場合でも、既存の `USE` を暗黙に `use once` へ変えるべきではない。必要であれば、include 型の `USE` とは別の明示的な機能として設計し、二重読み込み禁止・名前空間・可視性などの責務を分離して扱う。

## 表面構文と内部構造

TBX は外側では BASIC らしさを保つが、内側では小さな拡張可能コアへ寄せる。この二層性を意図的に維持する。

- ユーザーには、読みやすく実用的な BASIC 風構文を提供する
- 内部実装では、構文を小さな compile-time 操作と VM 操作へ還元する
- 高水準構文の見た目と、低レベル実装の直交性を無理に一致させない

ここで重要なのは、「表面構文が便利だから、その形をそのまま Rust の専用実装へ写す」と考えないことである。便利な表現は表面にあってよいが、内部では分解可能であることを優先する。

## Rust 側に追加してよいもの

Rust 側へ追加してよいのは、主に次の性質を持つものに限る。

- 複数の構文やライブラリ実装から再利用できる
- compile-time / runtime の責務境界を明確に保てる
- 特定構文名に依存せず説明できる
- 既存の抽象では不自然または過剰に複雑になる不足を埋める
- ソースコードを読むだけでは残りにくい規約を、実装として一貫して強制できる

逆に、避けるべきものは次の通りである。

- IF や FOR など特定構文 1 つのためだけの専用 primitive
- token / compile entry / runtime value の責務をまたぐ近道
- 一見便利でも、他の構文へ一般化できない ad hoc な分岐
- 標準ライブラリや IMMEDIATE ワードで十分表現できる糖衣を Rust 側へ固定化すること

## `VAR` 宣言のコンパイル方針

> Issue #636「VAR 宣言仕様を docs に反映する」に基づく設計方針

`VAR name = expr` の形式（宣言と初期化を同時に行う形式）は、Rust 側で直接実装している。これは特定構文への専用対応ではなく、次の理由による例外的な判断である。

ローカル宣言と初期化は、新しいローカル名を現在のワード定義のローカル領域へ追加したその直後に、そのローカルへ代入するという操作を同時に行う必要がある。この「登録してから即代入」という順序を既存の TBX 側 IMMEDIATE ワードだけで自然に表現するのは難しいため、Rust 側に置くことにした。

宣言のみの形式（`VAR name` / `VAR name, name, ...`）はローカルまたはグローバルのスロットを確保するだけで、実行時命令列を生成しない。これは TBX 側 IMMEDIATE ワードの仕組みで扱える範囲であり、特別扱いは必要としない。

トップレベルの `VAR name = expr` はエラーとする。グローバル変数の初期化を宣言構文に暗黙で紛れ込ませず、明示的な代入として書かせるためである。

## 追加判断チェックリスト

新しいコンパイル機能や compile-time primitive を追加するときは、少なくとも次を確認する。

1. その機能は IMMEDIATE ワードと既存語彙の組み合わせで表現できないか
2. Rust 側へ追加する場合、複数の構文から再利用できる抽象になっているか
3. 特定構文専用の名前や意味を持ち込んでいないか
4. `Cell`、`CompileEntry`、`Token` / `SpannedToken` の責務境界を壊していないか
5. Lexer、StatementReader、Compiler のどこに置くべき責務かを説明できるか
6. compile stack は汎用の compile-time 作業領域として使えているか
7. backpatching や前方参照解決を、個別構文の裏技ではなく一般操作として扱えているか
8. 表面構文の便利さを守りつつ、内部コアは小さく直交的なまま保てているか
9. `USE` のようなソース取り込み機能を変更するとき、include 型の再読み込み許可と循環検出の責務を混同していないか

## まとめ

TBX のコンパイル機構は、構文を増やすたびに Rust 側の特例を増やす方向ではなく、**小さなコア・明確な責務分離・IMMEDIATE ワード中心の拡張**へ寄せていくべきである。

この文書は、現在の実装詳細を保存するためではなく、今後の判断をぶらさないための設計原則文書として扱う。
