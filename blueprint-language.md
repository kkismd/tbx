# TBX コア言語仕様

> このファイルはTBX設計書の一部です。`blueprint.md`（VM・辞書のアーキテクチャ）および `blueprint-bootstrap.md`（ブートストラップ計画）と合わせて参照してください。

### コア言語の機能

#### 文法、字句構造

BASICの文法は複雑なので以下のように単純化する
* `ステートメント 単一の引数式 終端文字`の統一形式
    * 終端文字は改行かセミコロン
    * 引数式を評価する際、カンマを「左辺と右辺」を順にスタックに積む結合度の低い二項演算子とし、パラメータリストの代わりとする
* カンマを低優先度の二項演算子とするルールにより、`PLOT X, Y, COLOR` は内部的に `( (X , Y) , COLOR )` という一つの式として評価され、結果としてスタックに3つの値が順に積まれる。コマンド側は「スタックから必要な分だけ取り出す」だけで済む。

パースから実行まで

1. **スタック管理**: ベースポインタBPに現在のSPの値を保存する
1. **レキシカル分析:** 行を `[Command] [Rest of Line]` に分ける。
1. **式コンパイラ（ExprCompiler）:** `Rest of Line` を `ExprCompiler` が bytecode に変換し、インナー・インタプリタが実行する。このとき、カンマは「単に両辺をスタックに積む」という最低優先度の演算子とする。
1. **アリティ:** BPとSPの差から渡される引数の数を計算する。
1. **ディスパッチ:** `Command` に対応するルーチンを呼び出す。ルーチンはアリティを元にスタックから値を取り出して仕事をする。

#### 行番号と制御構造

BASICの全行に行番号を書く方式を廃止してFortranのように必要な箇所にだけローカルスコープの行番号を書いてGOTOの宛先を管理する。

GOSUB ~ RETURN によるサブルーチン呼び出しはなくし、手続きや関数は `DEF` ステートメントで新しいワードを定義することで作成する。
ステートメントは値を返すことができ、式の要素に使うことができる。
ワード定義の途中でワードから戻るには `RETURN` ステートメントを使う（詳細は「RETURN ステートメントとコンパイラによる分岐」を参照）。

**GOTO/BIF のスコープ制約**:

> Issue #116「ワード本体の終端判定機構（EXIT規約）が仕様に明記されていない」に基づく設計方針

`GOTO` / `BIF` / `BIT` のジャンプ先行番号は、そのワード本体内のローカル行番号に限定される。ワード本体外（別ワードの本体、またはトップレベル）へのジャンプを試みた場合はコンパイルエラーとする。この制約によりEXIT規約の完全性（ワード本体は必ず `EXIT` で終端する）が保証される。

#### 値の種類

> Issue #10「Cell型の定義が「値の種類」テーブルとRustコード例で不一致」に基づく設計方針

辞書やスタックのエレメント型であるCELLはホスト言語の共用体として定義される。(Rustのenum)

プリミティブ命令はすべてシステム辞書にワードとして登録し、`Xt`（実行トークン）を通じて呼び出す。これにより `OpCode` という専用バリアントは不要となり、Cell型はユーザーが扱う値の型のみで構成される。

> Issue #93「アドレス空間の統一: dictionary と data_stack が別々で Cell::Addr が曖昧」に基づく設計方針

| 型 (Variant)  | 内容 (Payload) | 用途                                                                                     |
| ------------- | -------------- | ---------------------------------------------------------------------------------------- |
| Int           | i64            | 通常の算術演算、ループカウンタ                                                           |
| Float         | f64            | 科学技術計算、座標計算                                                                   |
| DictAddr      | usize          | `dictionary[n]` を指すアドレス。グローバル変数・ヒープポインタ・TOKENディスクリプタ等   |
| StackAddr     | usize          | `data_stack[bp + n]` を指すアドレス。ローカル変数へのBPからのオフセット                 |
| Xt            | usize          | 実行可能なワードの開始アドレス                                                           |
| Bool          | bool           | 論理演算、IF の条件判定                                                                  |
| None          | -              | 空を表す値                                                                               |
| Array         | -              | 将来の拡張                                                                               |
| StringDesc    | usize          | 文字列プール内の先頭インデックス                                                         |
| Marker        | -              | データスタック上のsentinel。ステートメント境界を示す特殊値（Issue #151）                 |


#### コア言語でサポートされるステートメント

実装を進める中で他にも必要とされる機能があれば追加する。

基本命令

| ステートメント | 説明                                     |
| -------------- | ---------------------------------------- |
| LET            | 任意のアドレスへの値の書き込み（汎用 STORE のラッパー） |
| BIF            | 条件が不成立のときジャンプ (Branch If False)  |
| BIT            | 条件が成立のときジャンプ (Branch If True)     |
| GOTO           | 実行順の制御                             |
| PUTSTR         | 文字列リテラルを出力する（改行なし）     |
| PUTCHR         | ASCIIコードを数値として与えて1文字を出力する（改行なし） |
| PUTDEC         | 数値を10進フォーマットで出力する（改行なし） |
| PUTHEX         | 数値を16進フォーマットで出力する（改行なし） |
| HALT           | プログラムの停止                         |
| REM            | コメント                                 |

コア言語の条件分岐
* `BIF 条件式, ジャンプ先行番号` — 条件が**不成立**のときジャンプする。IF-ELSE-ENDIF等の構造化制御構文をコンパイルする際の基盤命令（条件不成立でELSE/ENDIF側に飛ぶ）として使う。
* `BIT 条件式, ジャンプ先行番号` — 条件が**成立**のときジャンプする。BASIC的な `IF expr THEN GOTO line` に相当する直感的な条件分岐。
* リッチな表現力を持つ条件分岐構文はあとで拡張構文として定義する。

```basic
REM 1から10までプリントする（BIT版: 条件成立でループ脱出）
DEF MYWORD
  10
      VAR I
      LET &I, 1
      PUTDEC I
      PUTSTR "\n"
      LET &I, I + 1
      BIT I > 10, 99
      GOTO 10
  99
END
```

```basic
REM FizzBuzz（コアステートメントのみで記述）
DEF FIZZBUZZ
      VAR I
      LET &I, 1
  10
        BIF (I % 3 = 0) & (I % 5 = 0), 20 ; REM 不成立の場合は20にジャンプ
          PUTSTR "FizzBuzz\n"
          GOTO 90
  20
        BIF I % 3 = 0, 30              ; REM 不成立の場合は30にジャンプ
          PUTSTR "Fizz\n"
          GOTO 90
  30
        BIF I % 5 = 0, 40              ; REM 不成立の場合は40にジャンプ
          PUTSTR "Buzz\n"
          GOTO 90
  40
          PUTDEC I
          PUTSTR "\n"
  90
      LET &I, I + 1
      BIT I <= 100, 10
END
```

上記の例ではコアステートメントのみを使用している。`PRINT`（値を出力して改行）や `FOR` / `NEXT`（ループ制御）は標準ライブラリ層で定義する予定であり、コア言語には含まない。

辞書操作

| ステートメント | 説明                             |
| -------------- | -------------------------------- |
| DEF            | ステートメントを定義             |
| VAR            | 変数の宣言                       |
| DIM            | グローバル配列の宣言（`DIM NAME(SIZE)` 形式）|
| END            | 定義の完了                       |
| HEADER         | 次のトークンを名前として辞書エントリを作成する |
| APPEND         | スタックトップのCellをdictionary[dp]に書き込みDPを進める |
| LITERAL        | スタックトップの値をLIT命令+値の2セルとして辞書に書き込む |
| ALLOT          | ヒープにN セルの領域を確保し、先頭アドレスを返す |
| HERE           | 現在の辞書ポインタ（次の書き込み先）をスタックに積む |

##### DIM — グローバル配列の宣言

`DIM NAME(SIZE)` 形式でグローバル配列を宣言する。辞書に `SIZE` セルの記憶領域を確保し、`NAME` を `EntryKind::Array { base, size }` として登録する。

```
DIM NAME(SIZE)    ( -- )
```

- トップレベル（実行モード）専用。`DEF..END` ブロック内での使用はエラー
- `SIZE` は正の整数リテラルのみ（0 以下はエラー）
- 辞書に `SIZE` 個の `Cell::None` を書き込み、先頭インデックスを `base` として記録
- `NAME(I)` で要素値の読み出し、`&NAME(I)` でアドレス取得（`OFFSET` 命令経由）
- 内部的に `OFFSET` 命令（`EntryKind::Offset`）が配列インデックスの境界チェックと DictAddr 計算を担う

```basic
DIM NUMS(5)       ' 5要素のグローバル配列を宣言
LET &NUMS(0), 42  ' NUMS[0] = 42 を格納
PUTDEC NUMS(0)    ' 42 を出力
```

##### HEADER — 辞書エントリの作成

ソースの直後のトークンを名前として読み取り、新しい `WordEntry` を `headers` に追加する。Forthの `CREATE` に相当する。

```
HEADER name    ( -- )
```

- ソースから次のトークンを名前として消費する（DEFと同じパターン）
- `headers` に新しい `WordEntry` を追加し、`latest` を更新する
- エントリの `kind` は `EntryKind::Word(dp)`（現在のDPをオフセットとして設定）
- `immediate` フラグは `false` で初期化される

##### APPEND — ワードのデータ領域に追加

スタックトップの `Cell` を `dictionary[dp]` に書き込み、DPを1進める。Forthの `,`（comma）に相当する汎用的なコンパイルプリミティブ。

```
APPEND    ( cell -- )
```

- スタックから `Cell` を1つポップする
- `dictionary[dp] = cell` に書き込む
- `dp += 1`
- `Cell::Xt`、`Cell::Int`、`Cell::DictAddr` 等、任意の `Cell` を書き込める

使用例: `LOOKUP` でXtを取得し、`APPEND` で辞書に書き込む。

```basic
REM "PUTSTR" のXtを辞書に書き込む
LOOKUP "PUTSTR"
APPEND
```

##### LITERAL — リテラル値のコンパイル

スタックトップの値を消費し、実行時にその値をスタックに積む命令列（2セル）を辞書に書き込む。Forthの `LITERAL` に相当する。

```
LITERAL    ( value -- )
```

- スタックから値を1つポップする
- `dictionary[dp]` に `Cell::Xt(LIT_xt)` を書き込む（`LIT` ワードのXt）
- `dictionary[dp+1]` にポップした値を書き込む
- `dp += 2`

LIT ワードの実行時動作:

```
LIT:
  push(dictionary[pc + 1])
  pc += 2
```

インナ・インタプリタが `LIT` のXtに到達すると、次のセル（`pc+1`）を値としてデータスタックに積み、`pc` を2進める。ここで `pc` はLIT自身のXtが格納されたセルを指している。通常の1セル命令では `pc += 1` だが、LITはインライン・オペランドを伴うため `pc += 2` となる。

パーサーと検索

| ステートメント | 説明                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------ |
| TOKEN          | ソースコードからトークンを切り出し、トークンの内部表現であるディスクリプタをスタックに積む |
| LOOKUP         | 辞書から検索してワードのXtをスタックに積む                                                 |
| COMPILE_EXPR   | ソースから式をパースし命令列をdictionary[dp..]に書き込む                                    |

##### COMPILE_EXPR — 式のコンパイル

ソースから式をパースし、SYA（操車場アルゴリズム）でRPN変換しながら命令列を `dictionary[dp..]` に直接書き込む。

```
COMPILE_EXPR    ( -- )
```

- ソースから式を1つ読み取る（トークンを消費する）
- 式を構成する各要素に応じた命令列を `dictionary[dp..]` に書き込み、DPを進める
- 実行時にその命令列を実行すると、式の評価結果がスタックに積まれる。カンマ演算子を含む式（例: `&I, I + 1`）では複数の値がスタックに残る

生成される命令列:

| 式の要素 | 生成される命令列 | セル数 |
| --- | --- | --- |
| 数値リテラル `42` | `Xt(LIT)`, `Int(42)` | 2 |
| 文字列リテラル `"hello"` | `Xt(LIT)`, `StringDesc(idx)` | 2 |
| グローバル変数 `A` | `Xt(LIT)`, `DictAddr(n)`, `Xt(FETCH)` | 3 |
| ローカル変数 `X` | `Xt(LIT)`, `StackAddr(n)`, `Xt(FETCH)` | 3 |
| リファレンス `&A`（グローバル） | `Xt(LIT)`, `DictAddr(n)` | 2 |
| リファレンス `&X`（ローカル） | `Xt(LIT)`, `StackAddr(n)` | 2 |
| 単項演算子 `-` | `Xt(NEGATE)` | 1 |
| 二項演算子 `+` | `Xt(ADD)` | 1 |
| ワード呼び出し `W(...)` | 引数列 + `Xt(CALL)`, `Xt(W)`, `Int(arity)`, `Int(local_count)` | 引数分 + 4 |
| プリミティブ/変数/定数呼び出し `P(...)` | 引数列 + `Xt(P)` | 引数分 + 1 |

> `CALL` は `EntryKind::Word` 専用。プリミティブ・変数・定数は直接 `Xt` を出力する。

トークン・ディスクリプタ

> Issue #153「トークンディスクリプタの保存場所」に基づく設計方針

`TOKEN` プリミティブは `parse_next_token()` を呼び出し、結果を3値としてスタックに積む。ヒープ（辞書）への書き込みは行わない。

```
TOKEN  ( -- kind len val )
```

| スタック位置 | Cell型 | 内容 |
| --- | --- | --- |
| `kind` | `Cell::Int` | トークン種別コード（下表参照） |
| `len`  | `Cell::Int` | 生ソースバイト長（文字列リテラルの場合はクォートとエスケープシーケンスを含む） |
| `val`  | `Cell::Int` または `Cell::DictAddr` | 数値リテラルの値、またはソースバッファ内のオフセット |

`val` の型はトークン種別によって決まる。数値リテラルなら `Cell::Int(i64)`、識別子・演算子などのソースバッファ参照なら `Cell::DictAddr(usize)` を使う。

`len` の具体例（全種別で生ソースバイト長に統一）:

| ソース入力 | `kind`    | `len` | 備考 |
| ---------- | --------- | ----- | ---- |
| `MYVAR`    | TOK_ID    | 5     | 識別子はそのまま |
| `42`       | TOK_NUM   | 2     | 数値リテラル |
| `+`        | TOK_OP    | 1     | 演算子 |
| `;`        | TOK_DELIM | 1     | 区切り文字 |
| `"hello"`  | TOK_STR   | 7     | クォート2文字を含む（`"` + `hello` + `"`） |
| `"\n"`     | TOK_STR   | 4     | クォート2文字 + バックスラッシュ + `n` |
| `"\\"`     | TOK_STR   | 4     | クォート2文字 + `\\` |

`val` が `Cell::DictAddr(ソースオフセット)` のトークン（TOK_ID・TOK_OP・TOK_DELIM・TOK_STR）は `source[val..val+len]` で生テキストを参照できる。TOK_NUM の `val` は `Cell::Int(解析済み整数値)` であり、ソースオフセットではない点に注意。

文字列リテラルを処理するTBXコードは、`val`（DictAddr）が指す生ソーステキストから自分でクォート除去とエスケープ展開を行う必要がある。

トークン種別コード（`kind` の値）

| 定数名    | 値 | 内容         | 例 |
| --------- | -- | ------------ | -- |
| TOK_ID    | 0  | 識別子       | `MYVAR`, `PRINT` |
| TOK_NUM   | 1  | 数値リテラル | `42`, `3.14` |
| TOK_OP    | 2  | 演算子       | `+`, `-`, `*`, `,` |
| TOK_DELIM | 3  | 区切り文字   | `;` |
| TOK_STR   | 4  | 文字列リテラル | `"hello"` |


##### DEF/END — ワード定義のコンパイル動作

> Issue #116「ワード本体の終端判定機構（EXIT規約）が仕様に明記されていない」に基づく設計方針

システムに「今は実行中か、それとも定義中か」を示すフラグ（is_compiling）を持たせて `DEF` を「即時実行ワード（IMMEDIATE）」として実装する。

DEF ワードの挙動
* フラグ is_compiling を true にする。
* 次のトークンを読み取り、それを「新しいワードの名前」として辞書に新規登録する。
* 名前に続く `(X, Y)` 形式の仮引数リストがあればパースし、局所変数テーブルをリセットしたうえで各引数名を `StackAddr(0)`, `StackAddr(1)`, ... として登録する（`arity = 引数の数`）。括弧がなければ `arity = 0`。
* 命令列の書き込み先は `dictionary[DP..]`（`DP_USER` 以降の空き領域）であり、DEF専用の別バッファは確保しない。コンパイルモード中、アウター・インタプリタはステートメント行を1行ずつ処理し、各ステートメントに対してフェーズ1と同じ命令列パターンを `dictionary[DP..]` に書き込む（DP を進めながら）。ステートメントの `EntryKind` に応じて2種類のパターンを使い分ける:

  **`EntryKind::Word` の場合:**
  ```
  Xt(LIT_MARKER)
  [引数式のコード]
  Xt(CALL), Xt(stmt), Int(arity), Int(0)
  Xt(DROP_TO_MARKER)
  ```

  **プリミティブ・変数・定数（それ以外の `EntryKind`）の場合:**
  ```
  Xt(LIT_MARKER)
  [引数式のコード]
  Xt(stmt)
  Xt(DROP_TO_MARKER)
  ```

VAR宣言はコンパイル時のローカル変数テーブルへの登録のみを行い、命令列は生成しない。`local_count` は VAR 宣言ごとに1増やす。RETURN / GOTO / BIF / BIT はそれぞれ対応する命令のXtを生成する。これを終わらせるのが END ワードです。

END ワードの挙動
* 命令列の末尾に EXIT のXtを書き込む（**EXIT規約**）。これがワード本体の終端判定の唯一のメカニズムであり、インナ・インタプリタは `EXIT` の実行によりワード本体の終端を検知する。
* `local_count = 局所変数テーブルのサイズ - arity` を確定し、この値をCALL命令のオペランドに埋め込む。
* 完成した命令列を辞書エントリに確定させる。
* フラグ is_compiling を false に戻す。

##### RETURN ステートメントとコンパイラによる分岐

> Issue #150「ワードから値を返す仕組み」に基づく設計方針

ユーザーが使える構文は `RETURN` 1種類だが、コンパイラが内部で2種類のプリミティブにコンパイルする。

| 構文 | コンパイル結果 | 動作 |
| ---- | -------------- | ---- |
| `RETURN expr` | 式の命令列 + `Xt(RETURN_VAL)` | 式を評価して戻り値をスタックに積み、RETURN_VAL で返す |
| `RETURN`（引数なし） | `Xt(EXIT)` | void returnとして即時リターン |
| `END`（ワード本体終端） | `Xt(EXIT)` | void returnとして終端する |

`RETURN expr` が式の途中（`f(x) + g(y)` など）で呼び出された場合も、RETURN_VAL の「退避 → truncate(bp) → 再push」によって引数・ローカル変数が正しく解放され、戻り値のみがスタックに残る。

複数の戻り値はサポートしない（単一値のみ）。

> Issue #128「コンパイル一時辞書のラベル・仮定義メカニズムが不明確」に基づく設計方針

**コンパイル進行中領域のライフサイクル**（`dictionary[DP_USER..DP]`）

DEF〜END 間の命令列は、辞書の独立した層ではなく、`DP_USER` 以降の空き領域（作業バッファ）に直接書き込む方式で管理する。

* **DEF 時**: ヘッダエントリを「未確定」状態で `headers` に追加し、命令列を `DP`（現在の書き込み先）以降に書き始める。`DP` は命令列の書き込みとともに進む。
* **END 時**: EXIT のXtを追記した後、`DP_USER = DP` として確定（マージ）する。ヘッダエントリの「未確定」状態を解除する。また、`RETURN expr` が出現した場合は対応する位置に `Xt(RETURN_VAL)` が書き込まれている。
* **エラー時**: `DP` を `DP_USER` に巻き戻し（書き込んだ命令列を破棄）、ヘッダエントリを削除する。

**行番号ラベルのコンパイラ内部管理**

行番号ラベル（`GOTO` / `BIF` / `BIT` のジャンプ先）はコンパイラ内部の Rust 構造体として管理する（辞書とは独立）。issue #117 のローカル変数テーブルと同じ設計パターン。

* `label_table: HashMap<i64, usize>` — 行番号 → `dictionary` インデックスのマッピング
* `patch_list: Vec<(i64, usize)>` — 未解決の行番号 → パッチすべき `dictionary` オフセットのリスト

DEF 開始時に両構造体を生成し、END 完了後に破棄する。行番号を辞書エントリとして登録しない（辞書の検索経路に行番号を混入させない）。

前方参照はバックパッチで解決する:

1. `GOTO N` コンパイル時にラベル `N` が未定義なら仮アドレス（`0`）を `dictionary` に書き込み、`patch_list` に `(N, offset)` を追記する。
2. ラベル `N` が出現したとき、現在の `DP` を `label_table[N]` として登録し、`patch_list` 内の `N` に対応するすべてのオフセットに正しいアドレスを書き戻す（バックパッチ）。
3. END 完了後に `patch_list` に未解決エントリが残っていた場合はコンパイルエラーとする。

#### ヒープへの書き込みを廃止する理由

旧仕様（`TOKEN ( -- addr )`）ではディスクリプタをヒープ上の3セル構造体として確保していたが、コンパイル中に `TOKEN` が呼ばれると descriptor cells が命令列に混入する「穴問題」が生じる。この問題を根本解消するため、スタック渡しに変更した。`label_table`・`patch_list`・ローカル変数テーブルと同じ設計パターン（VMまたはEvaluatorがコンパイラ内部状態をフィールドとして保持する方式）と一貫する。

#### 実装方針

- VM または Evaluator に `current_token: Option<TokenDescriptor>` フィールドを追加する
- `TOKEN` プリミティブは `parse_next_token()` を呼び出し、`kind`・`len`・`val` の順にスタックへ積む
- 辞書（ヒープ）には一切書き込まない

```
TOKEN              ( -- kind len val )
ROT TOK_ID =       ( len val flag )
```

種別チェック・値取得・名前検索はすべて3値スタックで達成できる。`Cell::Array` は将来の拡張として引き続き予約するが、コア言語の段階では使用しない。

##### IMMEDIATE — ワードに即時実行フラグを設定

ソースの直後のトークンをワード名として読み取り、そのワードの `FLAG_IMMEDIATE` フラグを立てる。

```
IMMEDIATE name    ( -- )
```

- ソースから次のトークンをワード名として消費する（HEADER と同じパターン）
- 辞書を検索し、指定されたワードの `flags` に `FLAG_IMMEDIATE` を設定する
- ワードが見つからない場合はエラーとする

この方式により、DEF〜END の直後に限らず、任意のタイミングで任意のワードに IMMEDIATE フラグを付与できる。Forth の `IMMEDIATE`（直前に定義されたワードに暗黙的に作用する）とは異なり、対象ワードを明示的に指定する TBX 独自の設計である。

> **実装上の注意**: IMMEDIATE はソース入力機構（トークン読み取り）に依存するため、HEADER と同様に VM 単体のプリミティブとしては実装できない可能性がある。アウターインタプリタ実装時にあわせて実装するか、issue #40 のスコープから除外する判断が必要である。

#### 式と演算子

* ステートメントとして呼び出す命令と、式の中で呼び出す関数は同じ辞書に定義されたワード。
* ステートメントが実行される前のSPを保存しておき、ステートメント実行中にSPが変化しても終了後に保存したSPに復帰する。
* ステートメントのパラメータとして記述される式の中ではワード呼び出しは`MYFUNC(1,2,3)`という形式で記述する。
* 式の評価の途中で即値や変数や関数呼び出しはスタックに値を積み、算術、比較演算子はスタックの値を消費する。

##### ステートメント境界のスタッククリア（マーカー方式）

ステートメントが返した値がデータスタックに残留するのを防ぐため、**マーカー方式**を採用する。各ステートメントのコンパイル時に `Cell::Marker` をsentinelとしてスタックに積み、ステートメント実行後に `DROP_TO_MARKER` でMarkerまで巻き戻す。

コンパイル後の命令列は以下の繰り返しになる:

```
LIT(Marker)            -- push statement boundary sentinel
... arg expression ... -- push arguments via expression evaluation
CALL(stmt, arity, 0)   -- invoke statement word
DROP_TO_MARKER         -- rewind stack to the sentinel (inclusive)
```

スタック遷移（arity=1、値を返さない場合）:

```
[B]        -- before
[B+1]      -- LIT(Marker)
[B+2]      -- push arg
CALL:      bp = B+1, EXIT -> truncate(B+1) -> [B+1]  (Marker remains)
DROP_TO_MARKER: -> [B]
```

スタック遷移（arity=1、値を返す場合）:

```
[B]        -- before
[B+1]      -- LIT(Marker)
[B+2]      -- push arg
CALL:      bp = B+1, RETURN_VAL -> truncate(B+1) -> push retval -> [B+2]
DROP_TO_MARKER: -> [B]  (retval also removed)
```

この方式を採用した理由:
- コンパイラがSPをコンパイル時に追跡する仕組みが不要（`ExprCompiler` はコンパイル時arity追跡カウンターを使用し、CALL命令のオペランドに静的に埋め込む）
- ワードの戻り型に関係なく普遍的に機能（辞書フラグやフロー分析が不要）
- ロジックが最もシンプルであり、メモリ効率・速度よりもシンプルさを優先する設計方針に合致する

追加が必要なもの:
- `Cell::Marker` — 新しいCellバリアント（データスタック上のsentinelとして識別可能な特殊値）
- `DROP_TO_MARKER` — 新プリミティブ（Markerを見つけるまでpopし続け、Markerも除去する）

##### 式評価器のモジュール構成

> Issue #87「操車場アルゴリズムでの関数適用について」に基づく設計方針

式コンパイル（SYAパーサー）は `expr.rs` として独立モジュールに分離する。式の評価は常に `ExprCompiler` がbytecodeを生成してインナー・インタプリタが実行する1パス方式とする。SYA処理中の関数呼び出しarity追跡カウンター（`frame_stack` 相当）はコンパイル中にのみ存在すればよい一時的な状態であり、VMの永続状態に追加するのは責務の混入になるため、`ExprCompiler` 構造体の内部状態として持つ。

| モジュール | 責務 |
| --- | --- |
| `vm.rs` | ランタイム状態（data_stack, return_stack, bp, 辞書）— 変更なし |
| `expr.rs`（新規） | 式のコンパイル（SYA, arity追跡カウンター, 演算子スタック） |

`ExprCompiler` 構造体は `VM` への可変参照を保持し、`compile_expr(&[SpannedToken]) -> Result<Vec<Cell>, TbxError>` で命令列を返す。即時実行専用の `Evaluator` 構造体は設計に含まない。

##### 式内関数呼び出しのarity確定方式

> Issue #87「操車場アルゴリズムでの関数適用について」に基づく設計方針

式内での関数呼び出し（例: `MYFUNC(1, 2, 3)`）のarity（引数の数）は**コンパイル時カウンター方式**で確定する（選択肢A）。`ExprCompiler` はSYA処理中にコンマをカウントし、`(` 検出時にarity=1でフレームを開き、内側のコンマが現れるたびにインクリメントする。arityはコンパイル時に確定し、**ワード呼び出し（`EntryKind::Word`）の場合のみ** CALL 命令のインラインオペランドとして bytecode に埋め込む。

処理の流れ:

1. `f(` を検出した時点で、演算子スタックに `LParen { call: Some((xt, arity: 1)) }` を積む
2. 引数はコンマを通じて順次コンパイルされる（コンマは引数区切りとして arity をインクリメント）
3. `)` で arity を確定し、呼び出し対象の `EntryKind` に応じて出力を分岐する:
   - `EntryKind::Word` の場合: `[Xt(CALL), Xt(f), Int(arity), Int(0)]` を出力キューに追加する
   - プリミティブ・変数・定数の場合: `[Xt(f)]` のみを出力キューに追加する（CALL なし）
4. 空の引数 `f()` の場合は arity=0 として同様に分岐して出力する

> Issue #194「blueprint-language.md：arity確定方式の記述（SP差分方式）がCOMPILE_EXPRテーブルと矛盾している」に基づく設計方針

> Issue #121「IMMEDIATEワード実行がframe_stackのarity計算を破壊する可能性」に基づく設計方針

**IMMEDIATEワードは式コンパイル中（`ExprCompiler` が関数呼び出しのarity追跡中、すなわち演算子スタックに `LParen { call: Some(..) }` が存在するとき）には使用できない。**

- コンパイラのIMMEDIATEワード実行処理において、実行前にarity追跡中でないかを確認する
- arity追跡中の場合（＝関数呼び出し引数のコンパイル中）は `CompileError::ImmediateInExpression` 相当のエラーを返す
- エラーメッセージは原因が明確に伝わる内容にする（例: "IMMEDIATEワードは式の中では使用できません"）
- 式の中でのマクロ的な引数注入ユースケースは、今後の要件として現れた時点で改めて検討する

この制約により、コンパイル時カウンター方式によるarity確定の設計一貫性が保たれる。IMMEDIATEワードの主な想定ユースケース（語彙切り替え・コンパイル設定変更）はステートメントレベルで使うものであり、関数呼び出し引数の中で使う場面は本来少ないため、問題領域は限定的である。

| 優先度 | 演算子               | 説明                               |
| ------ | -------------------- | ---------------------------------- |
| 0      | `(`, `)`, `.`        | 関数適用、構造体メンバ             |
| 1      | `-`, `!`, `&`        | 負符号、論理否定、リファレンス     |
| 2      | `*`, `/`, `%`        | 乗算、除算、剰余                   |
| 3      | `+`, `-`             | 加算、減算                         |
| 4      | `<`, `>`, `<=`, `>=` | 大小比較                           |
| 5      | `=`, `<>`            | 等号、不等号                       |
| 6      | `&`                  | ビット積                           |
| 7      | `|`                  | ビット和                           |
| 8      | `&&`                 | 論理積                             |
| 9      | `||`                 | 論理和                             |
| 10     | `?:`                 | 条件演算子                         |
| 11     | `,`                  | リスト演算子（スタックに値を残す） |

> Issue #12「演算子優先順位テーブルで `&` が単項・二項の両方に重複定義」に基づく設計方針

`-`（負符号 / 減算）と `&`（リファレンス / ビット積）は、同一の記号を単項・二項の両方で使用する。
パーサーは **直前のトークンの種類** によって単項・二項を判別する。これはCをはじめとする一般的な中置式パーサーの標準的な手法であり、Shunting Yard アルゴリズムに自然に組み込める。

| 直前のトークン | 演算子の解釈 |
| --- | --- |
| 式の先頭（トークンなし） | 単項（前置） |
| 演算子（`+` `-` `*` `/` など） | 単項（前置） |
| 開き括弧 `(` | 単項（前置） |
| カンマ `,` | 単項（前置） |
| 数値リテラル | 二項 |
| 識別子・変数名 | 二項 |
| 閉じ括弧 `)` | 二項 |

#### 変数

> Issue #11「EXIT命令がblueprint.mdに未定義」に基づく設計方針
> Issue #117「ローカル変数のメモリレイアウトと仮引数構文が未定義」に基づく設計方針
> Issue #120「LET &A, expr 構文がBASIC慣習から乖離している」に基づく設計方針

* グローバル変数はトップレベルのVARステートメントでグローバル辞書に登録される。（A-Zはプレリュードで登録され無宣言で使える）
* ローカル変数はコンパイル中のVARステートメントでローカル変数テーブルに登録され、実行時はスタックフレームにインデックスとして割当てられる。
* コンパイル時に名前をベースポインタからのインデックス番号に対応付けて実行時は名前が消える。
* DEFに書かれた仮引数と本体内で`VAR 変数名`で宣言したものがローカル変数名としてスタックに確保される。
* データスタックとは別にコールスタック（リターンスタック）を用意して戻り先の命令アドレスと呼び出し元のBPを保存する。ワード呼び出し時にBPを新しいフレームの先頭に設定し、EXITで元のBPを復元する。
* 変数の値を読み出す式は`A`、変数のリファレンスをとる式は`&A`。`&A` はコンパイル時にスコープを判定し、グローバル変数なら `DictAddr`、ローカル変数なら `StackAddr` を生成する。
* 変数への代入は`LET &A, A + 10`。

##### LET &A の設計意図

`LET` は「変数専用の代入命令」ではなく、**任意のアドレスに値を書き込む汎用 STORE のラッパー**として位置付ける。

```
LET &A, expr
```

- `&A` はアドレス式（`DictAddr` または `StackAddr` を返す）
- `expr` は書き込む値
- `LET` の実体は「`expr` を評価 → `&A` を評価 → `STORE`」の命令列

`&`（アドレス参照演算子）の明示的な指定はForthの設計哲学を継承したTBXのアイデンティティの一部である。コンパイラはスコープに基づいてアドレスを解決する。

| 変数の種類 | `&A` のコンパイル結果 |
| --- | --- |
| グローバル変数 | `DictAddr(n)` |
| ローカル変数 | `StackAddr(n)` |

`LET` が任意のアドレス式を受け付けるため、変数名以外のアドレス（ヒープポインタ等）への書き込みも同一構文で表現できる。

##### BASIC互換構文の非採用

`LET A, expr`（`&` 省略形）および `LET A = expr`（BASIC互換形）は採用しない。

- `&` を省略すると、左辺が変数名なのかアドレス式なのかが構文上で曖昧になる
- `LET A = expr` 形式はパーサを複雑にし、コアのミニマリズム原則に反する

TBXはBASICの方言ではなく、**BASICの見た目を持つForth影響下の新言語**である。`LET &A` における `&` の明示的なアドレス参照はその設計方針を体現する。`&` の意味（アドレス参照）はドキュメント・チュートリアルで丁寧に説明することで学習コストに対応する。

BASIC的な糖衣構文（`LET A = expr`）は現時点では提供しないが、将来の要件として現れた場合に改めて検討する。

##### 仮引数構文

DEFの仮引数は括弧付きリスト形式で記述する。呼び出し側の `MYFUNC(1, 2)` と対称的な構文になる。

```
DEF MYWORD(X, Y)   ' X = first arg, Y = second arg
  VAR Z            ' local variable
  ...
END
```

引数なしのワードは括弧を省略する。

```
DEF MYWORD
  ...
END
```

##### スコープ規則

各DEFは完全に独立した局所変数テーブルを持つ。コンパイル時にDEFの開始でテーブルをリセットし、仮引数名・VAR宣言名を `StackAddr(n)` に変換する。実行時には名前が消えるため、異なるDEF間での変数名の衝突は発生しない。

##### スタックフレームの初期化タイミング

CALL実行時にコンパイル時確定のローカル変数スロット数を `0` で事前確保する。VAR宣言はコンパイル時の名前登録のみを担い、実行時には何も行わない。

#### アドレス空間の設計方針

> Issue #93「アドレス空間の統一: dictionary と data_stack が別々で Cell::Addr が曖昧」に基づく設計方針

`dictionary`（`Vec<Cell>`）と `data_stack`（`Vec<Cell>`）は別々の配列として管理し、`Cell` のアドレスバリアントを2種類（`DictAddr(usize)` と `StackAddr(usize)`）に分けることで参照先を明示する（実装: `src/cell.rs`）。

`FETCH` / `STORE` はパターンマッチで参照先を切り替える。

```
FETCH:
  DictAddr(n)  => dictionary[n]
  StackAddr(n) => data_stack[bp + n]

STORE:
  DictAddr(n)  => dictionary[n]  <- value
  StackAddr(n) => data_stack[bp + n] <- value
```

従来の `Cell::Addr` を使っていた箇所は以下の方針で移行する。

| 用途 | 移行後のバリアント |
| --- | --- |
| `&A`（グローバル変数のリファレンス） | `DictAddr` |
| `&A`（ローカル変数のリファレンス） | `StackAddr` |
| TOKENディスクリプタのヒープポインタ | `DictAddr` |
| ディスクリプタ +2 フィールド（ソースバッファオフセット） | `DictAddr` |

##### スタックフレームのメモリレイアウト

> Issue #117「ローカル変数のメモリレイアウトと仮引数構文が未定義」に基づく設計方針

BP（Base Pointer）はフレームの先頭、すなわち **第1引数の位置** を指す。引数とローカル変数は連続して配置され、`StackAddr(n)` で統一アクセスする。

```
data_stack layout during a call to DEF FOO(X, Y) with VAR Z:

  index  StackAddr  value
  -----  ---------  -----
  BP+0   StackAddr(0)   X  -- 1st argument
  BP+1   StackAddr(1)   Y  -- 2nd argument
  BP+2   StackAddr(2)   Z  -- local variable (pre-filled with 0 at CALL time)
  ...
```

BPの設定と解放の手順:

> Issue #150「ワードから値を返す仕組み」に基づく設計方針

- **CALL実行時**:
  1. `vm.bp = vm.data_stack.len() - arity` でBPを設定（呼び出し側が引数をpushした後）
  2. `local_count` 個の `0` を data_stack にpushしてローカル変数スロットを確保
  3. 旧BP・リターンアドレスはコールスタック（data_stackとは独立した別領域）に退避する
- **EXIT実行時**（void return）:
  1. `vm.data_stack.truncate(vm.bp)` で引数・ローカル変数をまとめて解放する
  2. コールスタックからBPとPCを復元する
- **RETURN_VAL実行時**（値 return）:
  1. スタックトップの戻り値を一時退避する
  2. `vm.data_stack.truncate(vm.bp)` で引数・ローカル変数をまとめて解放する
  3. 戻り値を再pushする
  4. コールスタックからBPとPCを復元する

CALL命令はオペランドに `arity`（引数の数）と `local_count`（ローカル変数スロット数）の両方を埋め込む**4セル形式**を採用する:

```
[Xt(CALL), Xt(F), arity, local_count]  -- 4 cells total
```

`arity` と `local_count` は `Cell::Int` として埋め込み、CALL実行時に読み出す。

#### 文字列の扱いと出力命令

> Issue #2「コア言語における文字列の扱い」・Issue #4「数値を表示する基本命令」に基づく設計方針

コア言語の出力命令は以下の4つ。**すべて改行なし**で出力する。

| ステートメント | オペランド | 説明 |
| -------------- | ---------- | ---- |
| PUTSTR         | 文字列リテラル | 文字列を出力する |
| PUTCHR         | 数値式 | ASCIIコードに対応する1文字を出力する |
| PUTDEC         | 数値式 | 符号付き10進数で出力する |
| PUTHEX         | 数値式 | `$` プレフィックス付き大文字16進数で出力する |

改行が必要な場合は `PUTSTR` の文字列リテラル内でエスケープシーケンス `\n` を使うか、`PUTCHR 10` で改行文字を出力する。

##### PUTSTR のエスケープシーケンス

文字列リテラル内で以下のエスケープシーケンスを使用できる。コンパイル時に展開されて文字列プールに格納される。

| シーケンス | 意味 |
| ---------- | ---- |
| `\n`       | 改行 (LF, ASCII 10) |
| `\t`       | タブ (ASCII 9) |
| `\\`       | バックスラッシュ |

##### PUTDEC / PUTHEX の出力形式

- `PUTDEC`: 符号付き10進数。負数は `-` プレフィックス付き（例: `-42`, `100`）
- `PUTHEX`: `$` プレフィックス付き大文字16進数。最小桁数で出力（例: `$FF`, `$1000`）

##### 標準ライブラリによる拡張（改行付き命令）

改行を含む高レベルな出力命令は標準ライブラリ層で定義する。

| ステートメント | 説明 |
| -------------- | ---- |
| PRINTLN str    | PUTSTR str の後に改行を出力する |
| NEWLINE        | 改行のみを出力する（`PUTCHR 10` 相当） |

インタプリタ内部の文字列表現は以下の方針で実装する。

* 辞書データは `Cell` の配列で構成する。（`Cell` はRustのenumとして定義）
* 文字列データは辞書とは独立した **文字列プール（`Vec<u8>`）** に保存する。
* `Cell::StringDesc(usize)` のペイロードは文字列プール内の先頭インデックスを表す。
* **文字列プールは追記専用（append-only）**であり、単調増加する。`FORGET` 時も文字列プールは縮小しない（「FORGET とコンパクション」節を参照）。

##### 文字列プールのフォーマット

文字列プール内のエントリは **length-prefix形式** で連続して格納される。

| フィールド | サイズ | 内容 |
| ---------- | ------ | ---- |
| length | 2バイト（`u16` little-endian） | 後続データのUTF-8バイト長 |
| data | `length` バイト | UTF-8エンコードされた文字列データ |

**文字列リテラルの長さ上限: 65535バイト（UTF-8エンコード後）**

これはコア言語の字句仕様として規定する。パーサ（字句解析器）は文字列リテラルをスキャンした時点でUTF-8バイト長を検査し、65535バイトを超える場合はコンパイルエラーとして報告しなければならない。`intern_string` もこの上限を二重チェックする。実装の詳細は `src/cell.rs`・`src/vm.rs` を参照。

文字列プール内のフォーマット: 先頭2バイト（`u16` little-endian）に文字列長を格納し、続いてその長さ分の文字データを配置する（length-prefix方式）。

`PUTSTR` 実行時の動作:
1. `pc` が指す `Cell::StringDesc(idx)` を読み出す。
2. `string_pool[idx..idx+2]` から `u16` little-endian で長さを取得し、`string_pool[idx+2..]` から文字列データを読み出す。
3. 出力する（改行なし）。
4. `pc` を +1 する（すべての命令は1 Cell固定長）。

この設計により、すべての命令が **1 Cell 固定長** となり、実行ループがシンプルに保たれる。
静的な文字列のみを扱うため、GCは不要であり、プログラムのロード時に文字列プールを確定させる。
