# TBX プロジェクト

## 概要 (Overview)

Tiny BASIC のミニマリズムと Forth の動的な自己拡張性を融合させた処理系。最小限のコアプリミティブから出発し、言語自身を用いて段階的にブートストラップする。

eXtensibleなTiny BASICという意味で TBX という名前をつける。

### ドキュメントの方針

> Issue #74「blueprintのコード例がソースコードと重複している」に基づく設計方針
>
> 実装済みで安定したコンポーネントのRustコード例はblueprint.mdに記載しない。ソースコード（`src/` 以下）を正の実装定義とし、blueprint.mdには設計方針と仕様の記述に留める。

## アーキテクチャ (System Architecture)

システムの構造と開発段階はいくつかのレイヤーに分かれる
1. ホスト言語で記述されたVMとプリミティブ（ホスト言語で実装された最も基本的な操作。システム辞書にワードとして登録され、TBXコードから呼び出せる）、インナ・インタプリタ。
1. プリミティブを組み合わせて動作するコア言語
  1. Tiny BASIC相当の基本的なプログラム実行能力
  1. 自己拡張機能（基本的なワード定義機能とその部品であるコンパイルワード）
1. TBX自身で書かれた制御構造やステートメント

## 処理系の中核となるVM

* 辞書管理: 名前からワード定義を検索する仕組み。詳細は後述。
  * Xt（実行トークン: Execution Token）: 辞書に登録されたワードのエントリを参照するトークン（辞書配列のインデックス）。コンパイル済みワードの本体はXtの列として表現される。
* Cell（セル）: VMの最小メモリ単位。整数・浮動小数点数・アドレスなど複数の型を保持できるホスト言語の共用体として実装される。辞書とスタックはCellの配列として構成される。
* インナ・インタプリタ: Xtの列を順に読み、辞書を引いて実行するループ。
  * 間接スレッデッド・コード（ITC: Indirect Threaded Code）: ワードの本体をXtの列として表現し、インタプリタがXtを介して間接的にコードを辿って実行する方式。
* スタック
  * データスタック: 引数の受け渡しや演算を行う場所
    * SP（スタックポインタ）: データスタックの先頭を指すレジスタ
    * BP（ベースポインタ / フレームポインタ）: スタックフレーム（呼び出しごとに確保されるスタック上の固有領域）の境界を記録するレジスタ。ステートメント呼び出し時のSPをBPに保存し、ローカル変数はBPからのインデックスでアクセスする。BPとSPの差からアリティ（引数の数）を導出できる。
    * スタック操作プリミティブ: PUSH POP DROP LIT
  * リファレンス（アドレス）から値を読み書きする手段 FETCH STORE — `DictAddr` なら `dictionary[n]`、`StackAddr` なら `data_stack[bp + n]` を対象としてパターンマッチで分岐する
  * リターンスタック: 手続きからの戻り先アドレスと呼び出し元のBPを退避する。リターンスタックはシステム専用であり、ユーザーコードからの直接アクセス手段（Forthにおける `>R` / `R>` / `R@` 相当）は提供しない。TBXはワード定義に引数リストを持ち、CALL時にスタックフレームを構成するアーキテクチャのため、ローカル変数によりリターンスタックの一時退避用途が不要となる。
    * EXIT: リターンスタックから戻り先アドレスと元のBPを復元し、呼び出し元に制御を戻すプリミティブ
    * `ReturnFrame` 列挙型: リターンスタックに積まれるフレームの種別を表す型。以下の2バリアントを持つ。

> Issue #116「ワード本体の終端判定機構（EXIT規約）が仕様に明記されていない」に基づく設計方針

```rust
enum ReturnFrame {
    Call { return_pc: usize, saved_bp: usize }, // normal word call frame
    TopLevel,                                    // sentinel frame for top-level execution
}
```

**番哨フレーム方式（Sentinel Frame）**：トップレベル（「地の文」）のXt列実行開始前に `ReturnFrame::TopLevel` をリターンスタックに積む。インナ・インタプリタは `EXIT` 実行時に常にリターンスタックをpopし、取り出したフレームの種別で処理を分岐する。
* `ReturnFrame::Call` の場合: 保存された `return_pc` と `saved_bp` を復元して通常のリターンを行う。
* `ReturnFrame::TopLevel` の場合: 実行を正常終了する（これがトップレベル終端の唯一のメカニズムである）。
* リターンスタックが空の状態で `EXIT` が呼ばれた場合: 実装バグとして `StackUnderflow` エラーを返す。

> Issue #124「スタック深度制限（データスタック・リターンスタック）の仕様が未明記」に基づく設計方針

**スタック深度の上限値: `MAX_DATA_STACK_DEPTH = 65_536`（データスタック、65536セル）／ `MAX_RETURN_STACK_DEPTH = 4_096`（リターンスタック、4096フレーム）**

両値はコア言語の仕様として規定する定数であり、`MAX_DICTIONARY_CELLS` と同じスタイルでコア定数モジュールに定義する。

* データスタック：`vm.push()` メソッドでプッシュ前に `data_stack.len() >= MAX_DATA_STACK_DEPTH` をチェックする。
  - 超過時は `TbxError::DataStackOverflow { depth: usize, limit: usize }` を返す（既存の `TbxError::StringTooLong { len }` と同じスタイル）。
* リターンスタック：`CALL` 実行時に `return_stack.len() >= MAX_RETURN_STACK_DEPTH` をチェックする。
  - 超過時は `TbxError::ReturnStackOverflow { depth: usize, limit: usize }` を返す（同上）。
* いずれもスタックへの唯一の書き込み経路で1箇所のみチェックすれば十分である。

* 組み込みの自己拡張コンパイラ
  * 辞書の読み書きとポインタ: 辞書に新しいコードを書き込む手段。
  * レキシカルスキャナ: プログラムテキストを識別子、演算子、数値リテラル、文字列リテラルなどのトークンとして切り出す。
  * 辞書のルックアップ: ワード名から辞書を検索してワード定義を取り出す
  * 実行モード: 通常モードとコンパイルモードの切り替え
  * 即時実行ワード: コンパイルモードでもコンパイルされる代わりに実行するワード
  * パーザ: 優先度付きの演算子を含む式を操車場アルゴリズム（Shunting Yard Algorithm: 中置記法の式をスタックを使って後置記法に変換するアルゴリズム）でRPN（Reverse Polish Notation: 演算子を被演算子の後に置く記法。スタックマシンでの実行に適している）命令列に変換する。式評価（SYAパーサー・式実行）は `expr.rs` として独立モジュールに分離する（Issue #87）


### 辞書の構造と管理方針

> Issue #1「辞書の構造と管理方法」に基づく設計方針

#### ソースコードの管理

* **プリミティブ**はホスト言語（Rust）で実装され、ソースコードを持たない。
* **プリミティブ以外のワード**のソースコードは辞書内部には保持せず、**外部ファイルシステム上のソースファイル**として管理する。
* 辞書データはソースコードをコンパイルすることで構成される。

#### 辞書の階層構造

メモリ空間はひとつに統一し、論理的に以下の階層に分かれた辞書を優先度順に探索する。

| 層 | 辞書名 | 内容 |
| --- | --- | --- |
| 1 | システム辞書 | プリミティブを組み合わせて構成するコア言語の基本語彙 |
| 2 | 標準ライブラリ辞書 | TBXで記述された標準ライブラリ |
| 3 | ユーザー辞書 | ユーザーが定義したワード |
| 4 | コンパイル一時辞書 | コンパイル中のラベル・仮定義（完了後にマージまたは破棄） |

辞書の「境界」の管理
: メモリ空間内に**「各辞書の境界アドレス」を記録するシステム変数**（ポインタ）を持つ

* DP_SYS: システム辞書の終端
* DP_LIB: 標準ライブラリの終端
* DP_USER: ユーザー辞書の終端
* DP（辞書ポインタ / HERE）: 現在の書き込み先（DP_USER より先の空き領域の先頭）

#### 辞書エントリの構造（ヘッダとデータの分離）

> Issue #51「辞書エントリの管理方法について」に基づく設計方針

辞書は**ヘッダ層**と**データ層**を分離した二層構造で実装する。

| 層 | フィールド名 | 型 | 内容 |
| --- | --- | --- | --- |
| ヘッダ層 | `headers` | `Vec<WordEntry>` | ワード名・フラグ・エントリ種別を保持する構造体の配列 |
| データ層 | `dictionary` | `Vec<Cell>` | コンパイル済みコードをフラットに格納するCell配列 |

**ヘッダ層**はリンクリスト構造を持つ。各エントリに `prev: Option<usize>` フィールドを持たせ、`headers` 配列内の前エントリのインデックスを格納する。これにより以下を実現する。

* **シャドウイング**: 同名ワードを新たに登録する際、既存エントリを書き換えずに末尾に追加するだけでよい。検索は末尾（`latest`）から `prev` をたどって行うため、最新の定義が自動的に優先される。
* **FORGET による巻き戻し**: `latest` を特定のインデックスまで戻し、それ以降の `headers` エントリと `dictionary` データを破棄するだけで定義を取り消せる。

**データ層**はフラットな `Vec<Cell>` 配列であり、`pc` はこの配列へのインデックスとして機能する。DP_SYS / DP_LIB / DP_USER / DP もこの配列へのオフセットとして管理する。

`WordEntry` はエントリの種別を `EntryKind` enum として保持する。`Cell`（データスタックや変数の値）と `EntryKind`（辞書エントリの属性）を明確に分離することで、辞書の多様なエントリをひとつの型で統一的に扱える。実装の詳細は `src/dict.rs`・`src/cell.rs`・`src/vm.rs` を参照。

#### ヒープ

辞書ポインタ DP より先の未使用メモリ空間を**ヒープ**として扱う。`ALLOT` プリミティブで DP を進めることで領域を確保する。ヒープはユーザープログラムからも利用できる汎用領域であり、コンパイラが内部で一時データ（トークン・ディスクリプタなど）を置く場所でもある。

* ヒープのGCは行わない。`FORGET` や辞書の再コンパイル（コンパクション）によって不要領域を回収する。
* `ALLOT N` — DP を N セル分進めて領域を確保する。確保した領域の先頭アドレスを返す。
* `HERE` — 現在のDPの値（次の書き込み先アドレス）をスタックに積む。

> Issue #125「ヒープポインタDPのオーバーフロー対策が未定義」に基づく設計方針

**辞書の最大サイズ: `MAX_DICTIONARY_CELLS = 1_048_576`（1Mセル ≒ 8MB）**

これはコア言語の仕様として規定する定数であり、`STRING_MAX_LENGTH` と同じスタイルでコア定数モジュールに定義する。

* `ALLOT`・`APPEND`・`LITERAL` 等、DP を進めるすべての操作で DP 上限チェックを行う。
  - チェック方法: `let new_dp = dp + n; if new_dp > MAX_DICTIONARY_CELLS { return Err(TbxError::DictionaryOverflow { requested: new_dp, limit: MAX_DICTIONARY_CELLS }); }`
  - 超過時は `TbxError::DictionaryOverflow { requested: usize, limit: usize }` を返す（既存の `TbxError::StringTooLong { len }` と同じスタイル）。
* `ALLOT N` の引数 N が負値の場合は `TbxError::InvalidAllotCount` を返す。DP を後退させる操作は仕様として禁止する。

#### 永続化

* 辞書の永続化はメモリのダンプ（スナップショット保存）とリストア（ロード）によって行う。
* 実行中のメモリイメージをそのまま保存・復元することで高速な起動を実現する。

#### FORGET とコンパクション

> Issue #123「FORGET/コンパクション時の文字列プール参照の整合性が未定義」に基づく設計方針

* `FORGET` はユーザーの責任において行う。誤って重要なワードを削除した場合の復元は保証しない。
* `FORGET` は辞書（`headers` および `dictionary` 領域）のみをロールバックする。**文字列プール（string pool）は縮小しない**（追記専用・単調増加）。したがって `FORGET` 後も既存の `StringDesc` インデックスは有効なままであり、ダングリング参照は発生しない。
* システムや標準ライブラリの更新後は、外部ファイルシステム上のソースコードから辞書を**再コンパイル**し、メモリの洗替え（コンパクション）を行う。これにより断片化を解消し、整合性を保つ。
* **コンパクションは「辞書・文字列プール・ランタイム状態の全リセット」**である。コンパクション前にスタック上または変数内に保持されていた `StringDesc` 値は、コンパクション後に無効となる。コンパクションはユーザーが意図して行う重い操作として位置付ける。
* `FORGET` を繰り返すと文字列プールが徐々に増加するが、TBX の想定規模（組み込み系・小規模スクリプト）では実用上問題にならないため許容する。将来的に問題となる場合は別 issue で対処する。

### コア言語の機能

#### 文法、字句構造

BASICの文法は複雑なので以下のように単純化する
* `ステートメント 単一の引数式 終端文字`の統一形式
    * 終端文字は改行かセミコロン
    * 引数式を評価する際、カンマを「左辺と右辺」を順にスタックに積む結合度の低い二項演算子とし、パラメータリストの代わりとする
* カンマを低優先度の二項演算子とするルールにより、`PLOT X, Y, COLOR` は内部的に `( (X , Y) , COLOR )` という一つの式として評価され、結果としてスタックに3つの値が順に積まれる。コマンド側は「スタックから必要な分だけ取り出す」だけで済む。

パースから実行まで

1. **スタック管理**: ベースポインタBPに現在のSPの値を保存する
1. **レキシカル分析:** 行を `[Command] [Rest of Line]` に分ける。
1. **式評価器:** `Rest of Line` を評価する。このとき、カンマは「単に両辺をスタックに積む」という最低優先度の演算子とする。
1. **アリティ:** BPとSPの差から渡される引数の数を計算する。
1. **ディスパッチ:** `Command` に対応するルーチンを呼び出す。ルーチンはアリティを元にスタックから値を取り出して仕事をする。

#### 行番号と制御構造

BASICの全行に行番号を書く方式を廃止してFortranのように必要な箇所にだけローカルスコープの行番号を書いてGOTOの宛先を管理する。

GOSUB ~ RETURNで呼び出すサブルーチンをなくし、手続きや関数は新しいステートメントを定義することで作成する。
ステートメントは値を返すことができ、式の要素に使うことができる。

**GOTO/BIF のスコープ制約**:

> Issue #116「ワード本体の終端判定機構（EXIT規約）が仕様に明記されていない」に基づく設計方針

`GOTO` / `BIF` / `BIT` のジャンプ先行番号は、そのワード本体内のローカル行番号に限定される。ワード本体外（別ワードの本体、またはトップレベル）へのジャンプを試みた場合はコンパイルエラーとする。この制約によりEXIT規約の完全性（ワード本体は必ず `EXIT` で終端する）が保証される。

#### 値の種類

> Issue #10「Cell型の定義が「値の種類」テーブルとRustコード例で不一致」に基づく設計方針

辞書やスタックのエレメント型であるCELLはホスト言語の共用体として定義される。(Rustのenum)

プリミティブ命令はすべてシステム辞書にワードとして登録し、`Xt`（実行トークン）を通じて呼び出す。これにより `OpCode` という専用バリアントは不要となり、Cell型はユーザーが扱う値の型のみで構成される。

> Issue #93「アドレス空間の統一: dictionary と data_stack が別々で Cell::Addr が曖昧」に基づく設計方針

| 型 (Variant)  | 内容 (Payload) | 用途                                                                                     |
| ------------- | -------------- | ---------------------------------------------------------------------------------------- |
| Int           | i64            | 通常の算術演算、ループカウンタ                                                           |
| Float         | f64            | 科学技術計算、座標計算                                                                   |
| DictAddr      | usize          | `dictionary[n]` を指すアドレス。グローバル変数・ヒープポインタ・TOKENディスクリプタ等   |
| StackAddr     | usize          | `data_stack[bp + n]` を指すアドレス。ローカル変数へのBPからのオフセット                 |
| Xt            | usize          | 実行可能なワードの開始アドレス                                                           |
| Bool          | bool           | 論理演算、IF の条件判定                                                                  |
| None          | -              | 空を表す値                                                                               |
| Array         | -              | 将来の拡張                                                                               |
| StringDesc    | usize          | 文字列プール内の先頭インデックス                                                         |


#### コア言語でサポートされるステートメント

実装を進める中で他にも必要とされる機能があれば追加する。

基本命令

| ステートメント | 説明                                     |
| -------------- | ---------------------------------------- |
| LET            | 変数への代入                             |
| BIF            | 条件が不成立のときジャンプ (Branch If False)  |
| BIT            | 条件が成立のときジャンプ (Branch If True)     |
| GOTO           | 実行順の制御                             |
| PUTSTR         | 文字列リテラルを出力する（改行なし）     |
| PUTCHR         | ASCIIコードを数値として与えて1文字を出力する（改行なし） |
| PUTDEC         | 数値を10進フォーマットで出力する（改行なし） |
| PUTHEX         | 数値を16進フォーマットで出力する（改行なし） |
| HALT           | プログラムの停止                         |
| REM            | コメント                                 |

コア言語の条件分岐
* `BIF 条件式, ジャンプ先行番号` — 条件が**不成立**のときジャンプする。IF-ELSE-ENDIF等の構造化制御構文をコンパイルする際の基盤命令（条件不成立でELSE/ENDIF側に飛ぶ）として使う。
* `BIT 条件式, ジャンプ先行番号` — 条件が**成立**のときジャンプする。BASIC的な `IF expr THEN GOTO line` に相当する直感的な条件分岐。
* リッチな表現力を持つ条件分岐構文はあとで拡張構文として定義する。

```basic
REM 1から10までプリントする（BIT版: 条件成立でループ脱出）
DEF MYWORD
  10
      VAR I
      LET &I, 1
      PUTDEC I
      PUTSTR "\n"
      LET &I, I + 1
      BIT I > 10, 99
      GOTO 10
  99
END
```

```basic
REM FizzBuzz（コアステートメントのみで記述）
DEF FIZZBUZZ
      VAR I
      LET &I, 1
  10
        BIF (I % 3 = 0) & (I % 5 = 0), 20 ; REM 不成立の場合は20にジャンプ
          PUTSTR "FizzBuzz\n"
          GOTO 90
  20
        BIF I % 3 = 0, 30              ; REM 不成立の場合は30にジャンプ
          PUTSTR "Fizz\n"
          GOTO 90
  30
        BIF I % 5 = 0, 40              ; REM 不成立の場合は40にジャンプ
          PUTSTR "Buzz\n"
          GOTO 90
  40
          PUTDEC I
          PUTSTR "\n"
  90
      LET &I, I + 1
      BIT I <= 100, 10
END
```

上記の例ではコアステートメントのみを使用している。`PRINT`（値を出力して改行）や `FOR` / `NEXT`（ループ制御）は標準ライブラリ層で定義する予定であり、コア言語には含まない。

辞書操作

| ステートメント | 説明                             |
| -------------- | -------------------------------- |
| DEF            | ステートメントを定義             |
| VAR            | 変数の宣言                       |
| END            | 定義の完了                       |
| HEADER         | 次のトークンを名前として辞書エントリを作成する |
| APPEND         | スタックトップのCellをdictionary[dp]に書き込みDPを進める |
| LITERAL        | スタックトップの値をLIT命令+値の2セルとして辞書に書き込む |
| ALLOT          | ヒープにN セルの領域を確保し、先頭アドレスを返す |
| HERE           | 現在の辞書ポインタ（次の書き込み先）をスタックに積む |

##### HEADER — 辞書エントリの作成

ソースの直後のトークンを名前として読み取り、新しい `WordEntry` を `headers` に追加する。Forthの `CREATE` に相当する。

```
HEADER name    ( -- )
```

- ソースから次のトークンを名前として消費する（DEFと同じパターン）
- `headers` に新しい `WordEntry` を追加し、`latest` を更新する
- エントリの `kind` は `EntryKind::Word(dp)`（現在のDPをオフセットとして設定）
- `immediate` フラグは `false` で初期化される

##### APPEND — ワードのデータ領域に追加

スタックトップの `Cell` を `dictionary[dp]` に書き込み、DPを1進める。Forthの `,`（comma）に相当する汎用的なコンパイルプリミティブ。

```
APPEND    ( cell -- )
```

- スタックから `Cell` を1つポップする
- `dictionary[dp] = cell` に書き込む
- `dp += 1`
- `Cell::Xt`、`Cell::Int`、`Cell::DictAddr` 等、任意の `Cell` を書き込める

使用例: `LOOKUP` でXtを取得し、`APPEND` で辞書に書き込む。

```basic
REM "PUTSTR" のXtを辞書に書き込む
LOOKUP "PUTSTR"
APPEND
```

##### LITERAL — リテラル値のコンパイル

スタックトップの値を消費し、実行時にその値をスタックに積む命令列（2セル）を辞書に書き込む。Forthの `LITERAL` に相当する。

```
LITERAL    ( value -- )
```

- スタックから値を1つポップする
- `dictionary[dp]` に `Cell::Xt(LIT_xt)` を書き込む（`LIT` ワードのXt）
- `dictionary[dp+1]` にポップした値を書き込む
- `dp += 2`

LIT ワードの実行時動作:

```
LIT:
  push(dictionary[pc + 1])
  pc += 2
```

インナ・インタプリタが `LIT` のXtに到達すると、次のセル（`pc+1`）を値としてデータスタックに積み、`pc` を2進める。ここで `pc` はLIT自身のXtが格納されたセルを指している。通常の1セル命令では `pc += 1` だが、LITはインライン・オペランドを伴うため `pc += 2` となる。

パーサーと検索

| ステートメント | 説明                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------ |
| TOKEN          | ソースコードからトークンを切り出し、トークンの内部表現であるディスクリプタをスタックに積む |
| LOOKUP         | 辞書から検索してワードのXtをスタックに積む                                                 |
| COMPILE_EXPR   | ソースから式をパースし命令列をdictionary[dp..]に書き込む                                    |

##### COMPILE_EXPR — 式のコンパイル

ソースから式をパースし、SYA（操車場アルゴリズム）でRPN変換しながら命令列を `dictionary[dp..]` に直接書き込む。

```
COMPILE_EXPR    ( -- )
```

- ソースから式を1つ読み取る（トークンを消費する）
- 式を構成する各要素に応じた命令列を `dictionary[dp..]` に書き込み、DPを進める
- 実行時にその命令列を実行すると、式の評価結果がスタックに積まれる。カンマ演算子を含む式（例: `&I, I + 1`）では複数の値がスタックに残る

生成される命令列:

| 式の要素 | 生成される命令列 | セル数 |
| --- | --- | --- |
| 数値リテラル `42` | `Xt(LIT)`, `Int(42)` | 2 |
| 文字列リテラル `"hello"` | `Xt(LIT)`, `StringDesc(idx)` | 2 |
| グローバル変数 `A` | `Xt(LIT)`, `DictAddr(n)`, `Xt(FETCH)` | 3 |
| ローカル変数 `X` | `Xt(LIT)`, `StackAddr(n)`, `Xt(FETCH)` | 3 |
| リファレンス `&A`（グローバル） | `Xt(LIT)`, `DictAddr(n)` | 2 |
| リファレンス `&X`（ローカル） | `Xt(LIT)`, `StackAddr(n)` | 2 |
| 単項演算子 `-` | `Xt(NEGATE)` | 1 |
| 二項演算子 `+` | `Xt(ADD)` | 1 |
| 関数呼び出し `F(...)` | 引数列 + `Xt(CALL)`, `Xt(F)` | 引数分 + 2 |

トークン・ディスクリプタ

> Issue #14「トークン・ディスクリプタのValue/Ptrフィールドサイズが曖昧」に基づく設計方針

`TOKEN` ステートメントはディスクリプタをヒープ上の3セル構造体として確保し、その先頭アドレスを `Cell::DictAddr` として1値だけスタックに積む。コンパイラワードがディスクリプタを受け渡す際はこのアドレスを通じて `FETCH` / `STORE` でフィールドにアクセスする。

```
TOKEN  ( -- addr )   ; addr points to descriptor on heap
```

ヒープ上のディスクリプタ構造（3セル固定長）：

| オフセット | Cell型 | 内容 |
| --- | --- | --- |
| +0 | `Cell::Int` | トークン種別コード（下表参照） |
| +1 | `Cell::Int` | トークンの文字列長 |
| +2 | `Cell::Int` または `Cell::DictAddr` | 数値リテラルの値、またはソースバッファ内のオフセット |

`+2` フィールドの型はトークン種別によって決まる。数値リテラルなら `Cell::Int(i64)`、識別子・演算子などのソースバッファ参照なら `Cell::DictAddr(usize)` を使う。これにより「4-8 bytes」の曖昧さはなくなり、既存のCell型バリアントで表現できる。

トークン種別コード（オフセット +0 の値）

| 定数名    | 値 | 内容         | 例 |
| --------- | -- | ------------ | -- |
| TOK_ID    | 0  | 識別子       | `MYVAR`, `PRINT` |
| TOK_NUM   | 1  | 数値リテラル | `42`, `3.14` |
| TOK_OP    | 2  | 演算子       | `+`, `-`, `*`, `,` |
| TOK_DELIM | 3  | 区切り文字   | `;` |
| TOK_STR   | 4  | 文字列リテラル | `"hello"` |

ディスクリプタの寿命はコンパイルフェーズ内のみ。コンパイル完了後はヒープのコンパクション（FORGET等）により回収される。`Cell::Array` は将来の拡張として引き続き予約するが、コア言語の段階では使用しない。


状態管理と属性

| ステートメント | 説明                                           |
| -------------- | ---------------------------------------------- |
| STATE          | 実行モード=0, コンパイルモード=1               |
| IMMEDIATE      | オペランドを即時実行ワードとするフラグを立てる |



#### 式と演算子

* ステートメントとして呼び出す命令と、式の中で呼び出す関数は同じ辞書に定義されたワード。
* ステートメントが実行される前のSPを保存しておき、ステートメント実行中にSPが変化しても終了後に保存したSPに復帰する。
* ステートメントのパラメータとして記述される式の中ではワード呼び出しは`MYFUNC(1,2,3)`という形式で記述する。
* 式の評価の途中で即値や変数や関数呼び出しはスタックに値を積み、算術、比較演算子はスタックの値を消費する。

##### 式評価器のモジュール構成

> Issue #87「操車場アルゴリズムでの関数適用について」に基づく設計方針

式評価（SYAパーサー・式実行）は `expr.rs` として独立モジュールに分離する。`frame_stack`（式内関数呼び出しのarity計算用）は式評価中にのみ存在すれば良い一時的な状態であり、VMの永続状態に追加するのは責務の混入になるため、`Evaluator` 構造体のフィールドとして持つ。

| モジュール | 責務 |
| --- | --- |
| `vm.rs` | ランタイム状態（data_stack, return_stack, bp, 辞書）— 変更なし |
| `expr.rs`（新規） | 式のパース・評価（SYA, frame_stack, 演算子スタック） |

`Evaluator` 構造体は `VM` への可変参照・フレームスタック・演算子スタックを保持する。`vm.rs` の `bp: usize` はワード実行中のフレーム管理のために残し、変更しない。

##### 式内関数呼び出しのarity確定方式

> Issue #87「操車場アルゴリズムでの関数適用について」に基づく設計方針

式内での関数呼び出し（例: `MYFUNC(1, 2, 3)`）のarity（引数の数）は**実行時SP差分方式**で確定する。コンパイル時にコンマを数えるカウンター方式（選択肢A）は即時実行モードとコンパイルモードで処理パスが分岐するが、SP差分方式（選択肢B）はどちらのモードでも同一のスタック操作で動作し、BPを用いたステートメントのarity確定と一貫した設計になる。

処理の流れ:

1. `f(` を検出した時点で、`Evaluator` の `frame_stack` に現在のSP値を積む
2. 引数はコンマを通じて順次スタックに積まれる（コンマは最低優先度の演算子として機能する）
3. `)` で `CALL(f)` を実行し、arityは `current_sp - base_sp`（`frame_stack` からポップした値）で算出する
4. 空の引数 `f()` の場合、`frame_stack` からポップした直後のSPと現在のSPが等しいためarity=0となる

この方式は即時実行モードとコンパイルモードで同一メカニズムを使用し、可変引数関数も自然にサポートする。識別子の扱い（「識別子直後の `(` で関数呼び出し判定」か「レキサ段階でトークン化」か）は実装フェーズで検討する。

> Issue #121「IMMEDIATEワード実行がframe_stackのarity計算を破壊する可能性」に基づく設計方針

**IMMEDIATEワードは式評価中（`frame_stack` が非空のとき）には使用できない。**

- コンパイラのIMMEDIATEワード実行処理において、実行前に `frame_stack.is_empty()` を確認する
- `frame_stack` が非空の場合（＝関数呼び出し引数の評価中）は `CompileError::ImmediateInExpression` 相当のエラーを返す
- エラーメッセージは原因が明確に伝わる内容にする（例: "IMMEDIATEワードは式の中では使用できません"）
- 式の中でのマクロ的な引数注入ユースケースは、今後の要件として現れた時点で改めて検討する

この制約により、SP差分方式によるarity確定の設計一貫性が保たれる。IMMEDIATEワードの主な想定ユースケース（語彙切り替え・コンパイル設定変更）はステートメントレベルで使うものであり、関数呼び出し引数の中で使う場面は本来少ないため、問題領域は限定的である。

演算子の優先順位

| 優先度 | 演算子               | 説明                               |
| ------ | -------------------- | ---------------------------------- |
| 0      | `(`, `)`, `.`        | 関数適用、構造体メンバ             |
| 1      | `-`, `!`, `&`        | 負符号、論理否定、リファレンス     |
| 2      | `*`, `/`, `%`        | 乗算、除算、剰余                   |
| 3      | `+`, `-`             | 加算、減算                         |
| 4      | `<`, `>`, `<=`, `>=` | 大小比較                           |
| 5      | `=`, `<>`            | 等号、不等号                       |
| 6      | `&`                  | ビット積                           |
| 7      | `|`                  | ビット和                           |
| 8      | `&&`                 | 論理積                             |
| 9      | `||`                 | 論理和                             |
| 10     | `?:`                 | 条件演算子                         |
| 11     | `,`                  | リスト演算子（スタックに値を残す） |

> Issue #12「演算子優先順位テーブルで `&` が単項・二項の両方に重複定義」に基づく設計方針

`-`（負符号 / 減算）と `&`（リファレンス / ビット積）は、同一の記号を単項・二項の両方で使用する。
パーサーは **直前のトークンの種類** によって単項・二項を判別する。これはCをはじめとする一般的な中置式パーサーの標準的な手法であり、Shunting Yard アルゴリズムに自然に組み込める。

| 直前のトークン | 演算子の解釈 |
| --- | --- |
| 式の先頭（トークンなし） | 単項（前置） |
| 演算子（`+` `-` `*` `/` など） | 単項（前置） |
| 開き括弧 `(` | 単項（前置） |
| カンマ `,` | 単項（前置） |
| 数値リテラル | 二項 |
| 識別子・変数名 | 二項 |
| 閉じ括弧 `)` | 二項 |

#### 変数

> Issue #11「EXIT命令がblueprint.mdに未定義」に基づく設計方針
> Issue #117「ローカル変数のメモリレイアウトと仮引数構文が未定義」に基づく設計方針

* グローバル変数はトップレベルのVARステートメントでグローバル辞書に登録される。（A-Zはプレリュードで登録され無宣言で使える）
* ローカル変数はコンパイル中のVARステートメントでローカル変数テーブルに登録され、実行時はスタックフレームにインデックスとして割当てられる。
* コンパイル時に名前をベースポインタからのインデックス番号に対応付けて実行時は名前が消える。
* DEFに書かれた仮引数と本体内で`VAR 変数名`で宣言したものがローカル変数名としてスタックに確保される。
* データスタックとは別にコールスタック（リターンスタック）を用意して戻り先の命令アドレスと呼び出し元のBPを保存する。ワード呼び出し時にBPを新しいフレームの先頭に設定し、EXITで元のBPを復元する。
* 変数の値を読み出す式は`A`、変数のリファレンスをとる式は`&A`。`&A` はコンパイル時にスコープを判定し、グローバル変数なら `DictAddr`、ローカル変数なら `StackAddr` を生成する。
* 変数への代入は`LET &A, A + 10`。

##### 仮引数構文

DEFの仮引数は括弧付きリスト形式で記述する。呼び出し側の `MYFUNC(1, 2)` と対称的な構文になる。

```
DEF MYWORD(X, Y)   ' X = first arg, Y = second arg
  VAR Z            ' local variable
  ...
END
```

引数なしのワードは括弧を省略する。

```
DEF MYWORD
  ...
END
```

##### スコープ規則

各DEFは完全に独立した局所変数テーブルを持つ。コンパイル時にDEFの開始でテーブルをリセットし、仮引数名・VAR宣言名を `StackAddr(n)` に変換する。実行時には名前が消えるため、異なるDEF間での変数名の衝突は発生しない。

##### スタックフレームの初期化タイミング

CALL実行時にコンパイル時確定のローカル変数スロット数を `0` で事前確保する。VAR宣言はコンパイル時の名前登録のみを担い、実行時には何も行わない。

#### アドレス空間の設計方針

> Issue #93「アドレス空間の統一: dictionary と data_stack が別々で Cell::Addr が曖昧」に基づく設計方針

`dictionary`（`Vec<Cell>`）と `data_stack`（`Vec<Cell>`）は別々の配列として管理し、`Cell` のアドレスバリアントを2種類に分けることで参照先を明示する。

```rust
pub enum Cell {
    DictAddr(usize),   // dictionary[n] — global variables, heap pointers, TOKEN descriptors
    StackAddr(usize),  // data_stack[bp + n] — local variables (BP-relative offset)
    // ...
}
```

`FETCH` / `STORE` はパターンマッチで参照先を切り替える。

```
FETCH:
  DictAddr(n)  => dictionary[n]
  StackAddr(n) => data_stack[bp + n]

STORE:
  DictAddr(n)  => dictionary[n]  <- value
  StackAddr(n) => data_stack[bp + n] <- value
```

従来の `Cell::Addr` を使っていた箇所は以下の方針で移行する。

| 用途 | 移行後のバリアント |
| --- | --- |
| `&A`（グローバル変数のリファレンス） | `DictAddr` |
| `&A`（ローカル変数のリファレンス） | `StackAddr` |
| TOKENディスクリプタのヒープポインタ | `DictAddr` |
| ディスクリプタ +2 フィールド（ソースバッファオフセット） | `DictAddr` |

##### スタックフレームのメモリレイアウト

> Issue #117「ローカル変数のメモリレイアウトと仮引数構文が未定義」に基づく設計方針

BP（Base Pointer）はフレームの先頭、すなわち **第1引数の位置** を指す。引数とローカル変数は連続して配置され、`StackAddr(n)` で統一アクセスする。

```
data_stack layout during a call to DEF FOO(X, Y) with VAR Z:

  index  StackAddr  value
  -----  ---------  -----
  BP+0   StackAddr(0)   X  -- 1st argument
  BP+1   StackAddr(1)   Y  -- 2nd argument
  BP+2   StackAddr(2)   Z  -- local variable (pre-filled with 0 at CALL time)
  ...
```

BPの設定と解放の手順:

- **CALL実行時**:
  1. `vm.bp = vm.data_stack.len() - arity` でBPを設定（呼び出し側が引数をpushした後）
  2. `local_count` 個の `0` を data_stack にpushしてローカル変数スロットを確保
  3. 旧BP・リターンアドレスはコールスタック（data_stackとは独立した別領域）に退避する
- **EXIT実行時**:
  1. 戻り値があれば一時退避する
  2. `vm.data_stack.truncate(vm.bp)` で引数・ローカル変数をまとめて解放する
  3. 戻り値を再pushし、コールスタックからBPとPCを復元する

CALL命令のオペランドには `arity`（引数の数）と `local_count`（ローカル変数スロット数）の両方を埋め込む。

#### 文字列の扱いと出力命令

> Issue #2「コア言語における文字列の扱い」・Issue #4「数値を表示する基本命令」に基づく設計方針

コア言語の出力命令は以下の4つ。**すべて改行なし**で出力する。

| ステートメント | オペランド | 説明 |
| -------------- | ---------- | ---- |
| PUTSTR         | 文字列リテラル | 文字列を出力する |
| PUTCHR         | 数値式 | ASCIIコードに対応する1文字を出力する |
| PUTDEC         | 数値式 | 符号付き10進数で出力する |
| PUTHEX         | 数値式 | `$` プレフィックス付き大文字16進数で出力する |

改行が必要な場合は `PUTSTR` の文字列リテラル内でエスケープシーケンス `\n` を使うか、`PUTCHR 10` で改行文字を出力する。

##### PUTSTR のエスケープシーケンス

文字列リテラル内で以下のエスケープシーケンスを使用できる。コンパイル時に展開されて文字列プールに格納される。

| シーケンス | 意味 |
| ---------- | ---- |
| `\n`       | 改行 (LF, ASCII 10) |
| `\t`       | タブ (ASCII 9) |
| `\\`       | バックスラッシュ |

##### PUTDEC / PUTHEX の出力形式

- `PUTDEC`: 符号付き10進数。負数は `-` プレフィックス付き（例: `-42`, `100`）
- `PUTHEX`: `$` プレフィックス付き大文字16進数。最小桁数で出力（例: `$FF`, `$1000`）

##### 標準ライブラリによる拡張（改行付き命令）

改行を含む高レベルな出力命令は標準ライブラリ層で定義する。

| ステートメント | 説明 |
| -------------- | ---- |
| PRINTLN str    | PUTSTR str の後に改行を出力する |
| NEWLINE        | 改行のみを出力する（`PUTCHR 10` 相当） |

インタプリタ内部の文字列表現は以下の方針で実装する。

* 辞書データは `Cell` の配列で構成する。（`Cell` はRustのenumとして定義）
* 文字列データは辞書とは独立した **文字列プール（`Vec<u8>`）** に保存する。
* `Cell::StringDesc(usize)` のペイロードは文字列プール内の先頭インデックスを表す。
* **文字列プールは追記専用（append-only）**であり、単調増加する。`FORGET` 時も文字列プールは縮小しない（「FORGET とコンパクション」節を参照）。

##### 文字列プールのフォーマット

文字列プール内のエントリは **length-prefix形式** で連続して格納される。

| フィールド | サイズ | 内容 |
| ---------- | ------ | ---- |
| length | 2バイト（`u16` little-endian） | 後続データのUTF-8バイト長 |
| data | `length` バイト | UTF-8エンコードされた文字列データ |

**文字列リテラルの長さ上限: 65535バイト（UTF-8エンコード後）**

これはコア言語の字句仕様として規定する。パーサ（字句解析器）は文字列リテラルをスキャンした時点でUTF-8バイト長を検査し、65535バイトを超える場合はコンパイルエラーとして報告しなければならない。`intern_string` もこの上限を二重チェックする。実装の詳細は `src/cell.rs`・`src/vm.rs` を参照。

文字列プール内のフォーマット: 先頭2バイト（`u16` little-endian）に文字列長を格納し、続いてその長さ分の文字データを配置する（length-prefix方式）。

`PUTSTR` 実行時の動作:
1. `pc` が指す `Cell::StringDesc(idx)` を読み出す。
2. `string_pool[idx..idx+2]` から `u16` little-endian で長さを取得し、`string_pool[idx+2..]` から文字列データを読み出す。
3. 出力する（改行なし）。
4. `pc` を +1 する（すべての命令は1 Cell固定長）。

この設計により、すべての命令が **1 Cell 固定長** となり、実行ループがシンプルに保たれる。
静的な文字列のみを扱うため、GCは不要であり、プログラムのロード時に文字列プールを確定させる。

### ブートストラップ

* フェーズ 1：アウター・インタプリタ（外側）の構築
* フェーズ 2：実行状態状態管理フラグと DEF と END によるワード定義
* フェーズ 3：地の文の即時実行（インタプリタモードの完成）
* フェーズ 4：地の文のコンパイルとメインルーチン実行（フルプログラムモードの完成）

以下、各フェーズの詳細を説明する。

#### フェーズ1: アウター・インタプリタの構築

VM（インナ・インタプリタ）が「Xt（実行トークン）の列」を実行するのに対し、テキストを読み込んで命令を探すアウター・インタプリタの実装。
* トークナイズ: 入力文字列を空白で区切り、トークンを1つ取り出す。
* 辞書検索: そのトークンが辞書にあるか探す。
* 状態に応じた処理:
    * 実行モード: その単語のXtを取得して即実行する。
    * コンパイルモード: その単語のXtを、今作っている最中の命令列（`Vec<Cell::Xt>` 等）の末尾に追加する。
    * 即時実行ワード: コンパイル中もその場で実行される属性を持つワード。

#### フェーズ2: DEF/ENDによるワード定義

> Issue #116「ワード本体の終端判定機構（EXIT規約）が仕様に明記されていない」に基づく設計方針

システムに「今は実行中か、それとも定義中か」を示すフラグ（is_compiling）を持たせて `DEF` を「即時実行ワード（IMMEDIATE）」として実装する。

DEF ワードの挙動
* フラグ is_compiling を true にする。
* 次のトークンを読み取り、それを「新しいワードの名前」として辞書に新規登録する。
* 名前に続く `(X, Y)` 形式の仮引数リストがあればパースし、局所変数テーブルをリセットしたうえで各引数名を `StackAddr(0)`, `StackAddr(1)`, ... として登録する（`arity = 引数の数`）。括弧がなければ `arity = 0`。
* そのワードの「中身（命令列）」を格納するためのバッファ（Rustなら空の Vec）を用意する。

コンパイルモード中、アウター・インタプリタはひたすらXtをバッファに詰め込み続ける。VAR宣言が現れるたびに新しい `StackAddr(n)` を局所変数テーブルに追加する（`local_count` を1増やす）。これを終わらせるのが END ワードです。

END ワードの挙動
* バッファの最後に EXIT のXtを書き込む（**EXIT規約**）。これがワード本体の終端判定の唯一のメカニズムであり、インナ・インタプリタは `EXIT` の実行によりワード本体の終端を検知する。
* `local_count = 局所変数テーブルのサイズ - arity` を確定し、この値をCALL命令のオペランドに埋め込む。
* 完成した命令列を辞書エントリに確定させる。
* フラグ is_compiling を false に戻す。


#### フェーズ3: 地の文の即時実行

> Issue #13「フェーズ3の「地の文」実行方式が未決定」に基づく設計方針

インタプリタモードを完成させる。DEF...ENDブロック外の「地の文（即時実行ステートメント）」をトークン単位でその場で即時実行する。

* DEF...ENDブロックを見つけたらフェーズ2の手順でワードを辞書に登録する。
* ブロック外の地の文は、1ステートメントずつその場でXtを解決して即実行する。
* この段階ではプログラム全体を先読みしないため、後方で定義されたワードを地の文から呼ぶことはできない。
* REPL（対話型）モードはこのフェーズ3の動作と同等。

#### フェーズ4: 地の文のコンパイルとメインルーチン実行

フルプログラムモードを完成させる。ソースコードをシングルパスで処理し、地の文をメインルーチンとしてコンパイルしてから実行する。

* ソースをシングルパスで先頭から順に処理する（先読みなし）。
* DEFブロックは出現した時点で即座にコンパイルして辞書に登録する。後続のDEFはそれ以前に定義されたワードを使用できる。
* ブロック外の地の文はメインルーチンバッファに逐次コンパイルする（即実行はしない）。
* **IMMEDIATEワードは地の文のコンパイル中でもその場で実行される。** これにより、語彙切り替えやコンパイル設定の変更を行うワードを専用構文なしに実現できる（PRGMAワードという独立した概念は不要）。ただし、IMMEDIATEワードはステートメントレベル（式の外側）でのみ使用できる。式評価中（関数呼び出し引数の評価中）に検出した場合はコンパイルエラーとする（詳細は「式内関数呼び出しのarity確定方式」参照）。
* 全行を処理し終えた後、メインルーチンのXtをVMに渡して実行する。
* 地の文はすべてのDEFより後に実行されるため、地の文で変更した実行時の状態はDEFのコンパイルには影響しない。コンパイル時の振る舞いを変えたい場合はIMMEDIATEワードを使う。
