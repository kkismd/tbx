# TBX プロジェクト

## 概要 (Overview)

Tiny BASIC のミニマリズムと Forth の動的な自己拡張性を融合させた処理系。最小限のコアプリミティブから出発し、言語自身を用いて段階的にブートストラップする。

eXtensibleなTiny BASICという意味で TBX という名前をつける。

## アーキテクチャ (System Architecture)

システムの構造と開発段階はいくつかのレイヤーに分かれる
1. ホスト言語で記述されたVMとプリミティブ、インナ・インタプリタ。
1. プリミティブを組み合わせて動作するコア言語
  1. Tiny BASIC相当の基本的なプログラム実行能力
  1. 自己拡張機能（基本的なワード定義機能とその部品であるコンパイルワード）
1. TBX自身で書かれた制御構造やステートメント

### 処理系の中核となるVM

* 辞書管理: 名前からアドレス（またはバイトコード）を引くテーブル。
  * 複数の辞書を優先度順に探索することで実現する名前空間。
  * ラベル解決用の小さなテンポラリ辞書: コンパイル中のラベル位置を一時保存するもの。
  * 辞書はホスト言語の構造体として定義されるエントリヘッダと、連続するセルからなるメモリ空間。
  * エントリは名前やフラグと、メモリ空間に置かれたコード・データへのリンク（アドレス）を記録する。
* インナ・インタプリタ: トークンを順番に読み、辞書を引いて実行するループ。
  * 間接スレッデッド・コード（ITC: Indirect Threaded Code）
* スタック
  * データスタック: 引数の受け渡しや演算を行う場所
    * スタック操作プリミティブ: PUSH POP DROP LIT
  * リファレンス（アドレス）から値を読み書きする手段 FETCH STORE
  * 制御スタック: 手続きからの戻り先とデータスタックのベースポインタを退避する
* 組み込みの自己拡張コンパイラ
  * 辞書の読み書きとポインタ: 辞書に新しいコードを書き込む手段。
  * レキシカルスキャナ: プログラムテキストを識別子、演算子、数値リテラル、文字列リテラルなどのトークンとして切り出す。
  * 辞書のルックアップ: ワード名から辞書を検索してワード定義を取り出す
  * 実行モード: 通常モードとコンパイルモードの切り替え
  * 即時実行ワード: コンパイルモードでもコンパイルされる代わりに実行するワード
  * パーザ: 優先度付きの演算子を含む式を操車場アルゴリズム(Shunting Yard Algorithm)でRPN命令列に変換する


### 辞書の構造と管理方針

> Issue #1「辞書の構造と管理方法」に基づく設計方針

#### ソースコードの管理

* **プリミティブ**はホスト言語（Rust）で実装され、ソースコードを持たない。
* **プリミティブ以外のワード**のソースコードは辞書内部には保持せず、**外部ファイルシステム上のソースファイル**として管理する。
* 辞書データはソースコードをコンパイルすることで構成される。

#### 辞書の階層構造

メモリ空間はひとつに統一し、論理的に以下の階層に分かれた辞書を優先度順に探索する。

| 層 | 辞書名 | 内容 |
| --- | --- | --- |
| 1 | システム辞書 | プリミティブを組み合わせて構成するコア言語の基本語彙 |
| 2 | 標準ライブラリ辞書 | TBXで記述された標準ライブラリ |
| 3 | ユーザー辞書 | ユーザーが定義したワード |
| 4 | コンパイル一時辞書 | コンパイル中のラベル・仮定義（完了後にマージまたは破棄） |

辞書の「境界」の管理
: メモリ空間内に**「各辞書の境界アドレス」を記録するシステム変数**（ポインタ）を持つ

* DP_SYS: システム辞書の終端
* DP_LIB: 標準ライブラリの終端
* DP_USER: ユーザー辞書の終端

#### 永続化

* 辞書の永続化はメモリのダンプ（スナップショット保存）とリストア（ロード）によって行う。
* 実行中のメモリイメージをそのまま保存・復元することで高速な起動を実現する。

#### FORGET とコンパクション

* `FORGET` はユーザーの責任において行う。誤って重要なワードを削除した場合の復元は保証しない。
* システムや標準ライブラリの更新後は、外部ファイルシステム上のソースコードから辞書を**再コンパイル**し、メモリの洗替え（コンパクション）を行う。これにより断片化を解消し、整合性を保つ。

### コア言語の機能

#### 文法、字句構造

BASICの文法は複雑なので以下のように単純化する
* `ステートメント 単一の引数式 終端文字`の統一形式
    * 終端文字は改行かセミコロン
    * 引数式を評価する際、カンマを「左辺と右辺」を順にスタックに積む結合度の低い二項演算子とし、パラメータリストの代わりとする
* カンマを低優先度の二項演算子とするルールにより、`PLOT X, Y, COLOR` は内部的に `( (X , Y) , COLOR )` という一つの式として評価され、結果としてスタックに3つの値が順に積まれる。コマンド側は「スタックから必要な分だけ取り出す」だけで済む。

パースから実行まで

1. **スタック管理**: ベースポインタBPに現在のSPの値を保存する
1. **レキシカル分析:** 行を `[Command] [Rest of Line]` に分ける。
1. **式評価器:** `Rest of Line` を評価する。このとき、カンマは「単に両辺をスタックに積む」という最低優先度の演算子とする。
1. **アリティ:** BPとSPの差から渡される引数の数を計算する。
1. **ディスパッチ:** `Command` に対応するルーチンを呼び出す。ルーチンはアリティを元にスタックから値を取り出して仕事をする。

#### 行番号と制御構造

BASICの全行に行番号を書く方式を廃止してFortranのように必要な箇所にだけローカルスコープの行番号を書いてGOTOの宛先を管理する。

GOSUB ~ RETURNで呼び出すサブルーチンをなくし、手続きや関数は新しいステートメントを定義することで作成する。
ステートメントは値を返すことができ、式の要素に使うことができる。

#### コア言語でサポートされるステートメント

実装を進める中で他にも必要とされる機能があれば追加する。

基本命令

| ステートメント | 説明                                     |
| -------------- | ---------------------------------------- |
| LET            | 変数への代入                             |
| BIF            | 条件分岐 (Branch IF)                     |
| GOTO           | 実行順の制御                             |
| PUTSTR         | 文字列リテラルを出力する（改行なし）     |
| PUTCHR         | ASCIIコードを数値として与えて1文字を出力する（改行なし） |
| PUTDEC         | 数値を10進フォーマットで出力する（改行なし） |
| PUTHEX         | 数値を16進フォーマットで出力する（改行なし） |
| HALT           | プログラムの停止                         |
| REM            | コメント                                 |

コア言語の条件分岐
* `BIF 条件式,不成立の場合にジャンプする行番号`
* リッチな表現力を持つ条件分岐構文はあとで拡張構文として定義する。

```basic
REM 1から10までプリントする
DEF MYWORD
  10
      VAR I
      LET &I, 1
      PRINT I
      LET &I, I + 1
      BIF I > 10, 99
      GOTO 10
  99
END
```

```basic
DEF FIZZBUZZ
      VAR I
      FOR &I,1,100
        BIF (I % 3 = 0) & (I % 5 = 0), 10 ; REM 不成立の場合は10にジャンプ
          PRINT "FizzBuzz"
          GOTO 99
  10
        BIF I % 3 = 0, 20              ; REM 不成立の場合は20にジャンプ
          PRINT "Fizz"
          GOTO 99
  20
        BIF I % 5 = 0, 30              ; REM 不成立の場合は30にジャンプ
          PRINT "Buzz"
          GOTO 99
  30
          PRINT I
  99        
      NEXT &I
END
```

#### 式と演算子

* ステートメントとして呼び出す命令と、式の中で呼び出す関数は同じ辞書に定義されたワード。
* ステートメントが実行される前のSPを保存しておき、ステートメント実行中にSPが変化しても終了後に保存したSPに復帰する。
* ステートメントのパラメータとして記述される式の中ではワード呼び出しは`MYFUNC(1,2,3)`という形式で記述する。
* 式の評価の途中で即値や変数や関数呼び出しはスタックに値を積み、算術、比較演算子はスタックの値を消費する。

演算子の優先順位

| 優先度 | 演算子               | 説明                               |
| ------ | -------------------- | ---------------------------------- |
| 0      | `(`, `)`, `.`        | 関数適用、構造体メンバ             |
| 1      | `-`, `!`, `&`        | 負符号、論理否定、リファレンス     |
| 2      | `*`, `/`, `%`        | 乗算、除算、剰余                   |
| 3      | `+`, `-`             | 加算、減算                         |
| 4      | `<`, `>`, `<=`, `>=` | 大小比較                           |
| 5      | `=`, `<>`            | 等号、不等号                       |
| 6      | `&`                  | ビット積                           |
| 7      | `|`                  | ビット和                           |
| 8      | `&&`                 | 論理積                             |
| 9      | `||`                 | 論理和                             |
| 10     | `?:`                 | 条件演算子                         |
| 11     | `,`                  | リスト演算子（スタックに値を残す） |

> Issue #12「演算子優先順位テーブルで `&` が単項・二項の両方に重複定義」に基づく設計方針

`-`（負符号 / 減算）と `&`（リファレンス / ビット積）は、同一の記号を単項・二項の両方で使用する。
パーサーは **直前のトークンの種類** によって単項・二項を判別する。これはCをはじめとする一般的な中置式パーサーの標準的な手法であり、Shunting Yard アルゴリズムに自然に組み込める。

| 直前のトークン | 演算子の解釈 |
| --- | --- |
| 式の先頭（トークンなし） | 単項（前置） |
| 演算子（`+` `-` `*` `/` など） | 単項（前置） |
| 開き括弧 `(` | 単項（前置） |
| カンマ `,` | 単項（前置） |
| 数値リテラル | 二項 |
| 識別子・変数名 | 二項 |
| 閉じ括弧 `)` | 二項 |

#### 変数

* グローバル変数はトップレベルのVARステートメントでグローバル辞書に登録される。（A-Zはプレリュードで登録され無宣言で使える）
* ローカル変数はコンパイル中のVARステートメントでローカル変数テーブルに登録され、実行時はスタックフレームにインデックスとして割当てられる。
* コンパイル時に名前をベースポインタからのインデックス番号に対応付けて実行時は名前が消える。
* DEFに書かれた仮引数と本体内で`VAR 変数名`で宣言したものがローカル変数名としてスタックに確保される。
* データスタックとは別にリターンスタックを用意して戻り先の命令アドレスと元のベースポインタを保存する。
* 変数の値を読み出す式は`A`、変数のリファレンスをとる式は`&A`。
* 変数への代入は`LET &A, A + 10`。

辞書やスタックのエレメント型であるCELLはホスト言語の共用体として定義される。(Rustのenum)

| 型 (Variant) | 内容 (Payload) | 用途                               |

| Int          | i64            | 通常の算術演算、ループカウンタ     |
| Float        | f64            | 科学技術計算、座標計算             |
| Addr         | usize          | & で取得した変数の番地、メモリ操作 |
| Xt           | usize          | 実行可能なワードの開始アドレス     |
| Bool         | bool           | 論理演算、IF の条件判定            |
| None         | -              | 空を表す値                         |
| Array        | -              | 将来の拡張                         |
| StringDesc   | usize          | 文字列プール内の先頭インデックス   |


#### 文字列の扱いと出力命令

> Issue #2「コア言語における文字列の扱い」・Issue #4「数値を表示する基本命令」に基づく設計方針

コア言語の出力命令は以下の4つ。**すべて改行なし**で出力する。

| ステートメント | オペランド | 説明 |
| -------------- | ---------- | ---- |
| PUTSTR         | 文字列リテラル | 文字列を出力する |
| PUTCHR         | 数値式 | ASCIIコードに対応する1文字を出力する |
| PUTDEC         | 数値式 | 符号付き10進数で出力する |
| PUTHEX         | 数値式 | `$` プレフィックス付き大文字16進数で出力する |

改行が必要な場合は `PUTSTR` の文字列リテラル内でエスケープシーケンス `\n` を使うか、`PUTCHR 10` で改行文字を出力する。

##### PUTSTR のエスケープシーケンス

文字列リテラル内で以下のエスケープシーケンスを使用できる。コンパイル時に展開されて文字列プールに格納される。

| シーケンス | 意味 |
| ---------- | ---- |
| `\n`       | 改行 (LF, ASCII 10) |
| `\t`       | タブ (ASCII 9) |
| `\\`       | バックスラッシュ |

##### PUTDEC / PUTHEX の出力形式

- `PUTDEC`: 符号付き10進数。負数は `-` プレフィックス付き（例: `-42`, `100`）
- `PUTHEX`: `$` プレフィックス付き大文字16進数。最小桁数で出力（例: `$FF`, `$1000`）

##### 標準ライブラリによる拡張（改行付き命令）

改行を含む高レベルな出力命令は標準ライブラリ層で定義する。

| ステートメント | 説明 |
| -------------- | ---- |
| PRINTLN str    | PUTSTR str の後に改行を出力する |
| NEWLINE        | 改行のみを出力する（`PUTCHR 10` 相当） |

インタプリタ内部の文字列表現は以下の方針で実装する。

* 辞書データは `Cell` の配列で構成する。（`Cell` はRustのenumとして定義）
* 文字列データは辞書とは独立した **文字列プール（`Vec<u8>`）** に保存する。
* `Cell::StringDesc(usize)` のペイロードは文字列プール内の先頭インデックスを表す。

```rust
enum Cell {
    OpCode(OpCode),     // instruction
    Integer(i64),       // numeric value
    StringDesc(usize),  // index into string_pool
}

struct VM {
    dictionary: Vec<Cell>,
    string_pool: Vec<u8>, // all string data packed here
    pc: usize,
}
```

文字列プール内のフォーマット: 先頭1バイトに文字列長を格納し、続いてその長さ分の文字データを配置する（長さプレフィックス方式）。

`PUTSTR` 実行時の動作:
1. `pc` が指す `Cell::StringDesc(idx)` を読み出す。
2. `string_pool[idx]` から長さを取得し、`string_pool[idx+1..]` から文字列データを読み出す。
3. 出力する（改行なし）。
4. `pc` を +1 する（すべての命令は1 Cell固定長）。

この設計により、すべての命令が **1 Cell 固定長** となり、実行ループがシンプルに保たれる。
静的な文字列のみを扱うため、GCは不要であり、プログラムのロード時に文字列プールを確定させる。

### ブートストラップ

* フェーズ 1：アウター・インタプリタ（外側）の構築
* フェーズ 2：実行状態状態管理フラグと DEF と END によるワード定義
* フェーズ 3：プログラムの実行（ブートストラップの完成）

#### フェーズ1

VM（インナ・インタプリタ）が「命令IDの列」を実行するのに対し、テキストを読み込んで命令を探すアウター・インタプリタの実装。
* トークナイズ: 入力文字列を空白で区切り、トークンを1つ取り出す。
* 辞書検索: そのトークンが辞書にあるか探す。
* 状態に応じた処理:
    * 実行モード: その単語がネイティブなら即実行し、ユーザー定義ならVMの execute（前述の `DOCOL` 相当）に渡す。
    * コンパイルモード: その単語のIDを、今作っている最中の命令列（`Vec<usize>` 等）の末尾に追加する。
    * 即時実行ワード: コンパイル中もその場で実行される属性を持つワード。

#### フェーズ2

システムに「今は実行中か、それとも定義中か」を示すフラグ（is_compiling）を持たせて `DEF` を「即時実行ワード（IMMEDIATE）」として実装する。

DEF ワードの挙動
* フラグ is_compiling を true にする。
* 次のトークンを読み取り、それを「新しいワードの名前」として辞書に新規登録する。
* そのワードの「中身（命令列）」を格納するためのバッファ（Rustなら空の Vec）を用意する。

コンパイルモード中、アウター・インタプリタはひたすら命令IDをバッファに詰め込み続けます。これを終わらせるのが END ワードです。

END ワードの挙動
* バッファの最後に EXIT（戻り先スタックから復帰する命令）のIDを書き込む。
* 完成した命令列を辞書エントリに確定させる。
* フラグ is_compiling を false に戻す。

#### フェーズ3

「プログラムを実行する」という動作を「名前のない一時的なユーザー定義命令を作って実行する」ことで実現する。

* ソースコード全体を読み込む。
* DEF ... END ブロックを見つけたら、フェーズ2〜3の手順で辞書を増やす。
* ブロック外の「地の文（即時実行ステートメント）」は、その場で実行するか、あるいは一つの大きな「メインルーチン」としてコンパイルしてから、最後にそのメインルーチンをVMに実行させる。
