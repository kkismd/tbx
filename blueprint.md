# TBX プロジェクト

## 概要 (Overview)

Tiny BASIC のミニマリズムと Forth の動的な自己拡張性を融合させた処理系。最小限のコアプリミティブから出発し、言語自身を用いて段階的にブートストラップする。

eXtensibleなTiny BASICという意味で TBX という名前をつける。

### ドキュメントの方針

> Issue #74「blueprintのコード例がソースコードと重複している」に基づく設計方針
>
> 実装済みで安定したコンポーネントのRustコード例はblueprint.mdに記載しない。ソースコード（`src/` 以下）を正の実装定義とし、blueprint.mdには設計方針と仕様の記述に留める。

## アーキテクチャ (System Architecture)

システムの構造と開発段階はいくつかのレイヤーに分かれる
1. ホスト言語で記述されたVMとプリミティブ（ホスト言語で実装された最も基本的な操作。システム辞書にワードとして登録され、TBXコードから呼び出せる）、インナ・インタプリタ。
1. プリミティブを組み合わせて動作するコア言語
  1. Tiny BASIC相当の基本的なプログラム実行能力
  1. 自己拡張機能（基本的なワード定義機能とその部品であるコンパイルワード）
1. TBX自身で書かれた制御構造やステートメント

## 処理系の中核となるVM

* 辞書管理: 名前からワード定義を検索する仕組み。詳細は後述。
  * Xt（実行トークン: Execution Token）: 辞書に登録されたワードのエントリを参照するトークン（辞書配列のインデックス）。コンパイル済みワードの本体はXtの列として表現される。
* Cell（セル）: VMの最小メモリ単位。整数・浮動小数点数・アドレスなど複数の型を保持できるホスト言語の共用体として実装される。辞書とスタックはCellの配列として構成される。
* インナ・インタプリタ: Xtの列を順に読み、辞書を引いて実行するループ。
  * 間接スレッデッド・コード（ITC: Indirect Threaded Code）: ワードの本体をXtの列として表現し、インタプリタがXtを介して間接的にコードを辿って実行する方式。
* スタック
  * データスタック: 引数の受け渡しや演算を行う場所
    * SP（スタックポインタ）: データスタックの先頭を指すレジスタ
    * BP（ベースポインタ / フレームポインタ）: スタックフレーム（呼び出しごとに確保されるスタック上の固有領域）の境界を記録するレジスタ。ステートメント呼び出し時のSPをBPに保存し、ローカル変数はBPからのインデックスでアクセスする。BPとSPの差からアリティ（引数の数）を導出できる。
    * スタック操作プリミティブ: PUSH POP DROP LIT
  * リファレンス（アドレス）から値を読み書きする手段 FETCH STORE — `DictAddr` なら `dictionary[n]`、`StackAddr` なら `data_stack[bp + n]` を対象としてパターンマッチで分岐する
  * リターンスタック: 手続きからの戻り先アドレスと呼び出し元のBPを退避する。リターンスタックはシステム専用であり、ユーザーコードからの直接アクセス手段（Forthにおける `>R` / `R>` / `R@` 相当）は提供しない。TBXはワード定義に引数リストを持ち、CALL時にスタックフレームを構成するアーキテクチャのため、ローカル変数によりリターンスタックの一時退避用途が不要となる。
    * EXIT: void returnプリミティブ。`data_stack.truncate(bp)` で引数・ローカル変数を解放し、リターンスタックから戻り先アドレスと元のBPを復元して呼び出し元に制御を戻す
    * RETURN_VAL: 値returnプリミティブ（新設）。スタックトップを退避 → `data_stack.truncate(bp)` → 戻り値を再pushし、リターンスタックから戻り先アドレスと元のBPを復元して呼び出し元に制御を戻す
    * `ReturnFrame` 列挙型: リターンスタックに積まれるフレームの種別を表す型。`Call`（戻り先PC・保存BP・配列プールの長さスナップショット `saved_array_pool_len`・実引数個数 `actual_arity`）と `TopLevel`（番哨フレーム）の2バリアントを持つ（実装: `src/cell.rs`）。`saved_array_pool_len` は EXIT/RETURN_VAL 時に配列プールを巻き戻すために使用する。文字列は `Cell::Str(Rc<str>)` として参照カウントで管理されるため、フレーム退出時に巻き戻す文字列プールは存在しない。`actual_arity` は可変引数ワード内で `VA_COUNT` が参照する。

> Issue #116「ワード本体の終端判定機構（EXIT規約）が仕様に明記されていない」に基づく設計方針

**番哨フレーム方式（Sentinel Frame）**：トップレベル（「地の文」）のXt列実行開始前に `ReturnFrame::TopLevel` をリターンスタックに積む。インナ・インタプリタは `EXIT` 実行時に常にリターンスタックをpopし、取り出したフレームの種別で処理を分岐する。
* `ReturnFrame::Call` の場合: 保存された `return_pc` と `saved_bp` を復元して通常のリターンを行う。`vm.arrays` を `saved_array_pool_len` まで切り詰めてフレームローカルな配列を解放する。文字列は `Rc<str>` で管理されるため、フレーム退出時に追加の解放処理は不要である。
* `ReturnFrame::TopLevel` の場合: 実行を正常終了する（これがトップレベル終端の唯一のメカニズムである）。このとき `vm.global_array_pool_len = vm.arrays.len()` を更新し、トップレベルで生成された配列をグローバル扱いに昇格させる。
* リターンスタックが空の状態で `EXIT` が呼ばれた場合: 実装バグとして `StackUnderflow` エラーを返す。

> Issue #124「スタック深度制限（データスタック・リターンスタック）の仕様が未明記」に基づく設計方針

**スタック深度の上限値: `MAX_DATA_STACK_DEPTH = 65_536`（データスタック、65536セル）／ `MAX_RETURN_STACK_DEPTH = 4_096`（リターンスタック、4096フレーム）**

両値はコア言語の仕様として規定する定数であり、`MAX_DICTIONARY_CELLS` と同じスタイルでコア定数モジュールに定義する。

* データスタック：`vm.push()` メソッドでプッシュ前に `data_stack.len() >= MAX_DATA_STACK_DEPTH` をチェックする。
  - 超過時は `TbxError::DataStackOverflow { depth: usize, limit: usize }` を返す（既存の `TbxError::StringTooLong { len }` と同じスタイル）。
* リターンスタック：`CALL` 実行時に `return_stack.len() >= MAX_RETURN_STACK_DEPTH` をチェックする。
  - 超過時は `TbxError::ReturnStackOverflow { depth: usize, limit: usize }` を返す（同上）。
* いずれもスタックへの唯一の書き込み経路で1箇所のみチェックすれば十分である。

* 組み込みの自己拡張コンパイラ
  * 辞書の読み書きとポインタ: 辞書に新しいコードを書き込む手段。
  * レキシカルスキャナ: プログラムテキストを識別子、演算子、数値リテラル、文字列リテラルなどのトークンとして切り出す。
  * 辞書のルックアップ: ワード名から辞書を検索してワード定義を取り出す
  * 実行モード: 通常モードとコンパイルモードの切り替え
  * 即時実行ワード: コンパイルモードでもコンパイルされる代わりに実行するワード
  * パーザ: 優先度付きの演算子を含む式を操車場アルゴリズム（Shunting Yard Algorithm: 中置記法の式をスタックを使って後置記法に変換するアルゴリズム）でRPN（Reverse Polish Notation: 演算子を被演算子の後に置く記法。スタックマシンでの実行に適している）命令列に変換する。式評価（SYAパーサー・式実行）は `expr.rs` として独立モジュールに分離する（Issue #87）


### 辞書の構造と管理方針

> Issue #1「辞書の構造と管理方法」に基づく設計方針

#### ソースコードの管理

* **プリミティブ**はホスト言語（Rust）で実装され、ソースコードを持たない。
* **プリミティブ以外のワード**のソースコードは辞書内部には保持せず、**外部ファイルシステム上のソースファイル**として管理する。
* 辞書データはソースコードをコンパイルすることで構成される。

#### 辞書の階層構造

メモリ空間はひとつに統一し、論理的に以下の階層に分かれた辞書を優先度順に探索する。

| 層 | 辞書名 | 内容 |
| --- | --- | --- |
| 1 | システム辞書 | プリミティブを組み合わせて構成するコア言語の基本語彙 |
| 2 | 標準ライブラリ辞書 | TBXで記述された標準ライブラリ |
| 3 | ユーザー辞書 | ユーザーが定義したワード |

> Issue #128「コンパイル一時辞書のラベル・仮定義メカニズムが不明確」に基づく設計方針
>
> コンパイル一時辞書（旧第4層）は廃止した。DEF〜END 間のコンパイル進行中領域は辞書の独立した層としては管理せず、フェーズ2の作業バッファ（`dictionary[DP_USER..DP]`）として扱う。詳細はフェーズ2の記述を参照。

辞書の「境界」の管理
: メモリ空間内に**「各辞書の境界アドレス」を記録するシステム変数**（ポインタ）を持つ

* DP_SYS: システム辞書の終端
* DP_LIB: 標準ライブラリの終端
* DP_USER: ユーザー辞書の終端
* DP（辞書ポインタ / HERE）: 現在の書き込み先（DP_USER より先の空き領域の先頭）

#### 辞書エントリの構造（ヘッダとデータの分離）

> Issue #51「辞書エントリの管理方法について」に基づく設計方針

辞書は**ヘッダ層**と**データ層**を分離した二層構造で実装する。

| 層 | フィールド名 | 型 | 内容 |
| --- | --- | --- | --- |
| ヘッダ層 | `headers` | `Vec<WordEntry>` | ワード名・フラグ・エントリ種別を保持する構造体の配列 |
| データ層 | `dictionary` | `Vec<Cell>` | コンパイル済みコードをフラットに格納するCell配列 |

**ヘッダ層**はリンクリスト構造を持つ。各エントリに `prev: Option<usize>` フィールドを持たせ、`headers` 配列内の前エントリのインデックスを格納する。これにより以下を実現する。

* **シャドウイング**: 同名ワードを新たに登録する際、既存エントリを書き換えずに末尾に追加するだけでよい。検索は末尾（`latest`）から `prev` をたどって行うため、最新の定義が自動的に優先される。
* **コンパイル失敗時の内部ロールバック**: `DEF ... END` のコンパイルが失敗したときに、進行中の定義だけを取り消すために `latest` と `headers` / `dictionary` の末尾を巻き戻す。これは VM 内部で完結する機構であり、ユーザーから任意の既存定義を取り消す手段は提供しない。

**データ層**はフラットな `Vec<Cell>` 配列であり、`pc` はこの配列へのインデックスとして機能する。DP_SYS / DP_LIB / DP_USER / DP もこの配列へのオフセットとして管理する。

`WordEntry` はエントリの種別を `EntryKind` enum として保持する。`Cell`（データスタックや変数の値）と `EntryKind`（辞書エントリの属性）を明確に分離することで、辞書の多様なエントリをひとつの型で統一的に扱える。実装の詳細は `src/dict.rs`・`src/cell.rs`・`src/vm.rs` を参照。

#### ヒープ

辞書ポインタ DP より先の未使用メモリ空間を**ヒープ**として扱う。`ALLOT` プリミティブで DP を進めることで領域を確保する。ヒープはユーザープログラムからも利用できる汎用領域である。コンパイラの内部一時データ（トークン・ディスクリプタ等）はヒープではなく VM/Evaluator のフィールドとして保持する（issue #153 参照）。

* ヒープのGCは行わない。セッション中は単調増加し、不要領域の回収はソースからの再コンパイル（コンパクション）または VM の作り直しによるリセットで行う。
* `ALLOT N` — DP を N セル分進めて領域を確保する。確保した領域の先頭アドレスを返す。
* `HERE` — 現在のDPの値（次の書き込み先アドレス）をスタックに積む。

> Issue #125「ヒープポインタDPのオーバーフロー対策が未定義」に基づく設計方針

**辞書の最大サイズ: `MAX_DICTIONARY_CELLS = 1_048_576`（1Mセル ≒ 8MB）**

これはコア言語の仕様として規定する定数であり、`STRING_MAX_LENGTH` と同じスタイルでコア定数モジュールに定義する。

* `ALLOT`・`APPEND`・`LITERAL` 等、DP を進めるすべての操作で DP 上限チェックを行う。
  - チェック方法: `let new_dp = dp + n; if new_dp > MAX_DICTIONARY_CELLS { return Err(TbxError::DictionaryOverflow { requested: new_dp, limit: MAX_DICTIONARY_CELLS }); }`
  - 超過時は `TbxError::DictionaryOverflow { requested: usize, limit: usize }` を返す（既存の `TbxError::StringTooLong { len }` と同じスタイル）。
* `ALLOT N` の引数 N が負値の場合は `TbxError::InvalidAllotCount` を返す。DP を後退させる操作は仕様として禁止する。

#### 永続化

* 辞書の永続化はメモリのダンプ（スナップショット保存）とリストア（ロード）によって行う。
* 実行中のメモリイメージをそのまま保存・復元することで高速な起動を実現する。

#### セッションのライフサイクルとコンパクション

> Issue #540「FORGET を即廃止する」に基づく設計方針
>
> 過去には Forth 風の `FORGET` をユーザー機能として持っていたが、TBX はファイル実行を主な利用モデルとしており、対話的な辞書巻き戻しを公開機能として維持する価値が薄いため廃止した。任意時点まで辞書を巻き戻す手段はユーザーには提供しない。

* セッション中、辞書（`headers` / `dictionary`）は原則として**単調増加**する。ユーザーが任意の既存定義を取り消す手段は提供しない。
* 配列プール（`vm.arrays`）は「長寿命のエントリ」と「フレームローカルなエントリ」が混在する。トップレベルで生成された配列は長寿命となるが、呼び出しフレーム内で生成された配列は `EXIT` / `RETURN_VAL` 時に `ReturnFrame::Call` に保存された `saved_array_pool_len` まで truncate される。したがって、配列プールは単一フレーム視点では単調増加するが、セッション全体としては伸縮し得る。文字列（`Cell::Str`）は `Rc<str>` で参照カウント管理されるため、プール方式の寿命管理は不要である。`Cell::Str` はデータスタック・変数スロット・配列要素間で安全に共有でき、すべての参照が失われた時点で自動的に解放される。
* やり直しは以下のいずれかで行う。
  * **VM reset**: VM そのものを作り直して空のセッションから再開する。
  * **source reload / recompaction**: 外部ファイルシステム上のソースコードから辞書を再コンパイルし、メモリを洗い替える。
* **コンパクションは「辞書・配列プール・ランタイム状態の全リセット」**である。コンパクション前にスタック上または変数内に保持されていた `Cell::Str` / `Cell::Array` などのハンドルはコンパクション後に無効となる。文字列値は `Cell::Str(Rc<str>)` として表現される（issue #588、Phase 5B-D2）。コンパクションはユーザーが意図して行う重い操作として位置付ける。
* なお、`DEF ... END` のコンパイル失敗時には進行中の定義だけを取り消す内部ロールバックが走る。これは VM 内部の整合性維持機構であり、ユーザーから既存定義に対して任意に呼び出せるものではない。
