# TBX ブートストラップ計画

> このファイルはTBX設計書の一部です。`blueprint.md`（VM・辞書のアーキテクチャ）および `blueprint-language.md`（コア言語仕様）と合わせて参照してください。

### ブートストラップ

* フェーズ 1：アウター・インタプリタ（外側）の構築
* フェーズ 2：実行状態状態管理フラグと DEF と END によるワード定義
* フェーズ 3：地の文の即時実行（インタプリタモードの完成）
* フェーズ 4：地の文のコンパイルとメインルーチン実行（フルプログラムモードの完成）

#### 実行モードの概観

TBX の最終的な実行モデルは **2つのモードが共存** する形をとる。フェーズ4はフェーズ3を廃止・置き換えるものではない。

| モード | 対応フェーズ | 動作 | 用途 |
| --- | --- | --- | --- |
| インタプリタモード | フェーズ3 | ステートメントを1行ずつ即時実行 | REPL（対話型）・stdin パイプ |
| フルプログラムモード | フェーズ4 | ソース全体をコンパイル後にメインルーチンを実行 | ファイル実行（将来実装予定） |

**起動フロー（現在の実装）：**

```
tbx              → stdin を読み込み、行単位で即時実行（インタプリタモード）
tbx file.tbx     → ファイル全体をコンパイルしてメインルーチンを実行（フルプログラムモード）
```

以下、各フェーズの詳細を説明する。

#### フェーズ1: アウター・インタプリタの構築

TBX のアウター・インタプリタは、テキスト行をステートメント単位で処理し、インナ・インタプリタ（`vm.run()`）を通じて実行する。本フェーズでは **実行モードのみ** を実装する（コンパイルモードはフェーズ2で導入する）。

**行の読み込みと分解:**

1. 入力を1行ずつ読む。
2. 行の先頭トークン（識別子）を **ステートメント名** として取り出す。行番号が付く場合はスキップする。
3. 残りのテキストを **引数式** として扱う（空であってもよい）。

**ステートメントの実行:**

> Issue #200「blueprint-bootstrap.md: ステートメント実行パターンにプリミティブ向け仕様が欠落」に基づく設計方針

ステートメントを1つ実行するたびに、以下の命令列を一時バッファに構築し `vm.run()` で即実行する。
ステートメントの種別（`EntryKind`）によって命令列パターンが異なる。

**コンパイル済みワード（`EntryKind::Word`）向け:**

```
Xt(LIT_MARKER)                          // push statement boundary sentinel onto data stack
[引数式の命令列]                         // SYAパーサーが生成; カンマで複数引数が積まれる
Xt(CALL), Xt(stmt), Int(arity), Int(0)  // call statement word via CALL (4-cell form)
Xt(DROP_TO_MARKER)                      // unwind stack to sentinel
Xt(EXIT)                                // detect TopLevel sentinel frame and exit vm.run()
```

**プリミティブ・変数・定数（それ以外の `EntryKind`）向け:**

```
Xt(LIT_MARKER)                          // push statement boundary sentinel onto data stack
[引数式の命令列]                         // SYAパーサーが生成; カンマで複数引数が積まれる
Xt(stmt)                                // execute primitive/variable/constant directly
Xt(DROP_TO_MARKER)                      // unwind stack to sentinel
Xt(EXIT)                                // detect TopLevel sentinel frame and exit vm.run()
```

- ステートメント名を辞書で検索し、見つからなければエラー。
- `arity`（引数の数）はステートメントのシグネチャから確定する（例: `PUTDEC` は1、`SET` は2）。
- `Int(0)` は `local_count`（ステートメント呼び出し側はローカル変数を確保しない）。
- `CALL` 命令は `EntryKind::Word` 専用であるため、プリミティブ等に対して使用することはできない。
- `vm.run()` は `ReturnFrame::TopLevel` 番哨フレームを通じて `EXIT` 実行時に終了する。

#### フェーズ2: DEF/ENDによるワード定義

> Issue #116「ワード本体の終端判定機構（EXIT規約）が仕様に明記されていない」に基づく設計方針

システムに「今は実行中か、それとも定義中か」を示すフラグ（is_compiling）を持たせて `DEF` を「即時実行ワード（IMMEDIATE）」として実装する。


DEF/END のコンパイル動作詳細（DEF の挙動・EXIT 規約・RETURN のコンパイル・コンパイル進行中領域のライフサイクル・行番号ラベルのコンパイラ内部管理）については `blueprint-language.md` の「DEF/END — ワード定義のコンパイル動作」セクションを参照。

#### フェーズ3: 地の文の即時実行

> Issue #13「フェーズ3の「地の文」実行方式が未決定」に基づく設計方針

インタプリタモードを完成させる。DEF...ENDブロック外の「地の文（即時実行ステートメント）」をトークン単位でその場で即時実行する。

* DEF...ENDブロックを見つけたらフェーズ2の手順でワードを辞書に登録する。
* ブロック外の地の文は、1ステートメントずつフェーズ1で定めた2種類の命令列パターン（`EntryKind::Word` の場合は `LIT_MARKER → 式 → CALL Xt(stmt) Int(arity) Int(0) → DROP_TO_MARKER → EXIT`、それ以外の場合は `LIT_MARKER → 式 → Xt(stmt) → DROP_TO_MARKER → EXIT`）を一時バッファに構築し、`vm.run()` で即実行する。
* この段階ではプログラム全体を先読みしないため、後方で定義されたワードを地の文から呼ぶことはできない。
* REPL（対話型）モードはこのフェーズ3の動作と同等。

> **現在の実装状況（`main.rs`）：** `tbx` コマンドは引数なし（stdin）でフェーズ3の動作（`exec_line` による行単位逐次実行）、ファイル指定でフェーズ4の動作（`compile_program` によるフルプログラムモード）を使用している。

#### フェーズ4: 地の文のコンパイルとメインルーチン実行

フルプログラムモードを完成させる。ソースコードをシングルパスで処理し、地の文をメインルーチンとしてコンパイルしてから実行する。

* ソースをシングルパスで先頭から順に処理する（先読みなし）。
* DEFブロックは出現した時点で即座にコンパイルして辞書に登録する。後続のDEFはそれ以前に定義されたワードを使用できる。
* ブロック外の地の文は、フェーズ1で定めた2種類の命令列パターン（`EntryKind::Word` の場合は `LIT_MARKER → 式 → CALL Xt(stmt) Int(arity) Int(0) → DROP_TO_MARKER`、それ以外の場合は `LIT_MARKER → 式 → Xt(stmt) → DROP_TO_MARKER`）を使い、メインルーチン本体（`dictionary[DP..]`）に順次コンパイルする（即実行はしない）。全ステートメントのコンパイル後、最後に `Xt(EXIT)` を追記してメインルーチンを確定し、実行する。
* **IMMEDIATEワードは地の文のコンパイル中でもその場で実行される。** これにより、語彙切り替えやコンパイル設定の変更を行うワードを専用構文なしに実現できる（PRGMAワードという独立した概念は不要）。ただし、IMMEDIATEワードはステートメントレベル（式の外側）でのみ使用できる。式評価中（関数呼び出し引数の評価中）に検出した場合はコンパイルエラーとする（詳細は「式内関数呼び出しのarity確定方式」参照）。
* 全行を処理し終えた後、メインルーチンのXtをVMに渡して実行する。
* 地の文はすべてのDEFより後に実行されるため、地の文で変更した実行時の状態はDEFのコンパイルには影響しない。コンパイル時の振る舞いを変えたい場合はIMMEDIATEワードを使う。

> **地の文での行番号ラベルと GOTO の扱い（フルプログラムモード）：**
>
> | ケース | 動作 |
> |---|---|
> | 行番号ラベル（例：`10 PUTDEC 1` の `10`）in 地の文 | サイレント無視（BASIC互換の記法として許容） |
> | `GOTO N` in 地の文 | コンパイルエラー（`"GOTO outside DEF"`） |
>
> この非対称な挙動は `blueprint-language.md` の「GOTO/BIFのスコープ制約」（「ワード本体外へのジャンプはコンパイルエラー」）と整合する。
> 行番号ラベルを地の文に書いてもGOTOジャンプ先として機能しない点に注意すること。

> **フェーズ3との関係：** フェーズ4はフェーズ3（インタプリタモード）を廃止しない。フェーズ4実装後は「ファイル実行時はフルプログラムモード（フェーズ4）、REPL/stdinはインタプリタモード（フェーズ3）」という形で両モードが共存する。
